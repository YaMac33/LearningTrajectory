NoSQLとは？RDBとの違いと4つの代表的なデータモデル

#データベース #NoSQL #ビッグデータ #基本情報 #応用情報

「Not Only SQL」の略称であり、従来のリレーショナルデータベース（RDB）の限界を補うために登場した新しいデータベースの総称。それが「NoSQL」です。大量のデータを高速かつ柔軟に扱うための仕組みを解説します。

【なぜNoSQLが生まれたのか？】
従来のRDBは「データの正確さ（整合性）」や「複雑な検索」には強いものの、「データの形がバラバラ」だったり、「アクセスが爆発的に多い」ビッグデータの処理は苦手でした。そこをカバーするために作られたのがNoSQLです。

1. RDBとNoSQLの決定的な違い（スケーリング）

システムを拡張して性能を上げる際、RDBとNoSQLでは得意なアプローチが全く異なります。

RDB（従来型）
スケールアップ（性能向上）が得意。
データが複雑に絡み合っているため、サーバーの台数を増やす（分散させる）のが難しく、1台のサーバーを高価な部品で強くするアプローチになりがちです。
NoSQL
➡
スケールアウト（台数増加）が得意。
構造がシンプルなため、データを複数台のサーバーに分割して配置することが容易です。安価なサーバーを並べることで、巨大なデータ量に対応できます。

2. NoSQLの4つの代表的なデータモデル

NoSQLと一口に言っても、データの持ち方（モデル）によって大きく4種類に分類されます。試験でもそれぞれの特徴と代表的な製品名が問われます。

🔑 キー・バリュー型
"Key1" : "値データ"
"Key2" : "値データ"
単純な「キー（名前）」と「バリュー（値）」のペアで保存する最もシンプルな形式。処理速度が圧倒的に速いのが特徴です。
代表例：Redis, Amazon DynamoDB
📄 ドキュメント型
{
  "name": "Taro",
  "skills": ["Java", "AWS"]
}
JSONやXMLなどの「ドキュメント形式」で保存します。データの形（スキーマ）を後から自由に変更できる柔軟性が強みです。
代表例：MongoDB, CouchDB
📊 カラム指向型
[ID] [名前] [年齢]
↑列(カラム)単位で管理
RDBが行（行データ）ごとに保存するのに対し、列（カラム）ごとにデータをまとめて保存します。特定の列だけを高速に集計するのに向いています。
代表例：Apache Cassandra, HBase
🕸️ グラフ型
(A) ─[友達]➡ (B)
(A) ─[いいね]➡ (C)
データ（ノード）と、その「つながり（エッジ）」の関係性を表現することに特化したデータベースです。SNSの人間関係分析などで活躍します。
代表例：Neo4j, Amazon Neptune

3. データの一貫性（ACID vs BASE）

RDBが厳密な整合性（ACID特性）を重視するのに対し、多くのNoSQLは「BASE特性」という考え方を採用しています。

結果整合性（Eventual Consistency）の考え方
「データが更新された瞬間は、他のサーバーで古いデータが見えてしまうかもしれないけど、"最終的（Eventual）には" 全てのサーバーでデータが一致するから良しとしよう」という割り切った考え方です。
この妥協により、分散処理時の圧倒的なスピードと可用性（システムが止まらないこと）を実現しています。

試験対策まとめ：キーワードで見抜く

モデル名 試験問題でのキーワード
キー・バリュー型 一意のキーと値のペア、高速な読み書き、インメモリ
ドキュメント型 JSON、XML、スキーマレス（構造が自由）
カラム指向型 列（カラム）単位、大量データの集計・分析向き
グラフ型 ノード（点）とエッジ（線）、関係性、SNSのつながり
