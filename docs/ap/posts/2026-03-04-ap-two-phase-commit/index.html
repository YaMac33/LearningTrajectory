2相コミット（2PC）とは？分散トランザクションの原子性と一貫性

#データベース #トランザクション #2相コミット #基本情報 #応用情報

複数のデータベースやサービスをまたいで処理を行う「分散システム」において、絶対にデータの矛盾を起こさないための仕組みが「2相コミット（Two-Phase Commit：2PC）」です。

1. なぜ必要なのか？（原子性と一貫性の課題）

トランザクション処理において最も重要なのは「全部成功するか、全部失敗するかのどちらか（原子性：Atomicity）」であることです。
これが守られないと、データに致命的な矛盾（一貫性の崩壊）が生じます。

銀行振込における「部分的な失敗」の恐怖
理想（全部成功）
DB1：A口座 -10,000円
⬇
DB2：B口座 +10,000円
整合性が保たれる
問題（途中で失敗）
DB1：A口座 -10,000円
⬇ ❌通信エラー等
DB2：B口座 +0円
1万円が消滅する！（矛盾）

2. 2相コミットの登場人物

この問題を解決するため、2PCでは全体を仕切るリーダーと、作業を行うメンバーに役割を分けます。

👑 調停者 (Coordinator)
全体を管理し、最終決定を下す
参加者
(DB 1)
参加者
(DB 2)
参加者
(DB 3)

3. 2つの段階（Phase1 / Phase2）の流れ

名前の通り、処理を「準備」と「決定」の2段階に分けて行います。

Phase 1：Prepare（準備フェーズ）
Coordinator
コミットできる？ (Prepare)
Participants
Coordinator
YES! / NO...
Participants

各DBに準備確認を行い、参加者は結果を返します（YESならPrepared状態）。

Phase 2：Commit（決定フェーズ）
【全員がYESだった場合】
Coordinator
全員 COMMIT せよ！
全DB 反映
【1人でもNOだった場合】
Coordinator
全員 ROLLBACK せよ！
全DB 取消

全員の準備が確認できるまで絶対にコミットしないため、「一部だけ成功してしまう」という事態を防ぐことができます。

4. 2相コミットの欠点（ブロッキング問題）

安全性を高めるための2PCですが、致命的な弱点が存在します。

ブロッキング（処理の停止）
Phase1で「YES」を返した直後に...
💥 Coordinator が故障してダウン
指示が来ない！ ⬇
Participant（参加者）は待機状態
(勝手に判断できないため、データにロックをかけたまま止まる)

試験対策：ACID特性との関係

2相コミットは、トランザクションの4つの特性（ACID）のうち、特に以下の2つを強力に保証するための仕組みです。

要素 意味 2PCの貢献度
A Atomicity (原子性) ◎ (主目的) 全成功か全失敗か
C Consistency (一貫性) ◎ (主目的) 矛盾を作らない
I Isolation (独立性) ○ (ロック機構等で対応)
D Durability (永続性) ○ (ログへの書き出し等で対応)
