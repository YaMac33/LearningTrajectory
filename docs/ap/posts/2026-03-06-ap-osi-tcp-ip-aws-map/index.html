【図解】OSI 7階層・TCP/IP・Webアクセス・AWSの対応マップ

#ネットワーク #AWS #インフラ #トラブルシューティング

ネットワークの理論である「OSI参照モデル」と「TCP/IP」。これらが実際のWebアクセスにおいてどのような役割を果たし、さらにクラウドインフラ（AWS）のどのサービスがそのレイヤーを担っているのか。4つの軸を1枚のマップに統合して全体像を俯瞰します。

💡 なぜこの全体像が必要なのか？
「Webサイトの接続が遅い」「エラーが出る」といったトラブルが発生した際、この階層マップが頭に入っていると、「Route 53（L3）のDNS解決の問題か？」「ELB（L4/L7）の負荷が高いのか？」といった原因の切り分けがスムーズに行えるようになります。

1. ネットワーク階層とAWSサービスの対応マップ

上層（アプリケーション側）から下層（物理的な通信側）へ向かって、各要素がどのように対応しているかを確認しましょう。

2. マップから読み解く3つの重要ポイント

🧩 OSIとTCP/IPの「境界線の違い」
OSIモデルの上位3層（第5〜7層）は、TCP/IPモデルではまとめて「アプリケーション層」として扱われます。
AWSにおいても、CloudFrontやALB（Application Load Balancer）がこの領域（HTTPリクエスト処理、SSL暗号化、セッション維持）を一手に引き受ける設計になっています。
🌍 AWSの強み（L3/L4の最適化）
通常、オンプレミスでは物理的なケーブル（L1）やルーターの設定（L3）を深く意識する必要があります。
しかしAWSでは、Route 53やGlobal Acceleratorを利用することで、AWSが持つ世界中のエッジロケーションと専用線網を使って「自動的に最短経路」を選択してくれます。

3. トラブルシューティングの基本：「カプセル化」の逆算

データは送信される際、上位層から下位層へ向かってヘッダ（荷札）が付けられていきます（カプセル化）。受信側では逆に、下位層から上位層へ向かって荷札を剥がして処理します。

🔍 トラブルシューティングの実践

AWSにデータが届いた際、サーバーは「L1（物理） ➡ L3（IP） ➡ L4（TCP） ➡ L7（HTTP）」の順にデータを解いていきます。
もし通信トラブルが起きた場合、この順番通りに「下から上へ（ボトムアップ）」確認するのが鉄則です。
「VPCのルーティング（L3）は正しいか？」 ➡ 「セキュリティグループ（L4のポート）は開いているか？」 ➡ 「ALBのターゲット（L7）は正常か？」と順を追うことで、確実な原因特定が可能になります。
