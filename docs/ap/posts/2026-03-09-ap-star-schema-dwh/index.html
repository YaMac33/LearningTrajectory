スタースキーマとは？データウェアハウスの基本構造と特徴

#データベース #データウェアハウス #BIツール #基本情報 #応用情報

業務システムで蓄積されたデータを、BIツールなどで「分析・集計」しやすくするために整理する設計手法。その代表格であり、データウェアハウス（DWH）の基本モデルとなるのが「スタースキーマ（Star Schema）」です。

【1行要約】
スタースキーマとは、分析用データベースにおいて、中心となる「ファクトテーブル（事実・数値）」の周囲に、分析の切り口となる「ディメンションテーブル（属性）」を星型（放射状）に配置したデータ構造のことです。

1. なぜ「スター（星型）」なのか？

システムを構成するテーブル群を、「計測された実績値」と「それを分析するための軸」という2種類に明確に分離し、キーで結合する構造をとります。図にすると星のように見えるのが特徴です。

2. 2種類のテーブルの役割（ファクト vs ディメンション）

試験では、この2つのテーブルがそれぞれ「どんなデータを持つか」が頻出します。

📊 ファクトテーブル
  • 目的： 数値データ（測定値・実績）を集約して保存する。
  • 特徴： 日々データが蓄積されるため、データ量が非常に膨大になる。
  • 構造： ディメンションテーブルと繋がるための「キー（ID）」をたくさん持っている。
【具体例】
売上金額、販売数量、アクセス数、各ID（商品ID、顧客ID等）
🏷️ ディメンションテーブル
  • 目的： データを集計・分析するための「切り口（属性情報）」を提供する。
  • 特徴： ファクトテーブルの外側に複数配置される。マスターデータに近い性質で、データ量は少なめ。
  • 構造： 正規化をあえて崩して（冗長化して）、検索速度を優先することがある。
【具体例】
商品（名前、カテゴリ）、顧客（氏名、地域、年齢層）、日付（年、月、曜日）

3. スタースキーマのメリットと実務での使われ方

通常のデータベース（RDB）では、データの重複をなくすため細かくテーブルを分割（正規化）しますが、スタースキーマでは「分析するときのクエリ（SQL）をシンプルにして、集計を高速化する」ことを最優先します。

実務のSQLイメージ（カテゴリ別の売上集計）
SELECT d.カテゴリ, SUM(f.売上金額)
FROM ファクト_売上 f
JOIN ディメンション_商品 d ON f.商品ID = d.商品ID
GROUP BY d.カテゴリ;

中央のファクトと周囲のディメンションを1回 JOIN（結合）するだけで、欲しい分析結果がすぐに出せます。
