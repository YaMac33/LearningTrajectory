「制約チェックのタイミング」と「チェックポイント」の違い

#データベース #トランザクション #基本情報 #応用情報

情報処理試験のデータベース分野で、名前の響きが似ているために間違えやすい「制約チェック」と「チェックポイント」。全く異なる2つの概念を整理して解説します。

【よくある問題パターン】
「データベースにおいて、制約の検査を行う時点を指定できる機能はどれか？」

この問題文を見たとき、名前に「チェック」と入っているからといって「チェックポイント」を選んでしまうと誤りになります。
これは「即時制約（Immediate）と遅延制約（Deferred）」に関する説明です。

1. そもそも「制約（Constraint）」とは？

データベースに間違ったデータ（矛盾したデータ）が入らないようにするためのルールのことです。
代表的なものに、重複を許さない「主キー制約」や、他のテーブルに存在しない値を入れさせない「外部キー制約」などがあります。

2. 整合性チェックのタイミング（即時 vs 遅延）

この「ルール違反していないか？」を確認するタイミングには、2つの方式があります。

トランザクション内のチェックタイミング
即時制約 (Immediate)
SQL 1🔍チェック
➡
SQL 2🔍チェック
➡
SQL 3🔍チェック
➡
COMMIT

各SQL文が実行される直後にその都度チェックする方式。

遅延制約 (Deferred)
SQL 1（保留）
➡
SQL 2（保留）
➡
SQL 3（保留）
➡
COMMIT
🚨 まとめてチェック

途中のSQLでは違反を許容し、コミットの瞬間に最終確認する方式。

3. なぜ「遅延制約」が必要なのか？

全て即時チェックで良いように思えますが、以下のような「相互に参照し合っているテーブル」にデータを追加する際に困ったことが起きます。

遅延制約が必要になるケース（相互参照）
テーブル A
➡ 参照 ➡ ⬅ 参照 ⬅
テーブル B

Aを入れようとすると「Bがないからダメ（外部キー制約違反）」と怒られ、先にBを入れようとすると「Aがないからダメ」と怒られます（デッドロック状態）。
遅延制約にしておけば、とりあえず両方とも入れてしまい、「最後に両方存在していればOK」として解決できます。

4. 「チェックポイント」との決定的な違い

問題文の「検査時点を指定する」という表現は上記の即時・遅延の話ですが、これと「チェックポイント」は全く別の機能です。

🛡️ 制約チェック 目的：データの正当性を守る
おかしなデータが入らないようにルール確認をすること。
タイミングを「各SQLごと（即時）」か「コミット時（遅延）」かで選べる。
💾 チェックポイント 目的：障害回復を高速化する
メモリ（バッファ）上にある更新データを、ディスクにまとめて書き出す機能。
クラッシュした際に、ここから復旧を始めれば良いという「しおり」の役割。

まとめ表

用語 目的・役割 キーワード
制約チェック (即時/遅延) データ整合性の維持 検査時点、コミット時、相互参照
チェックポイント 障害からの回復時間を短縮 ディスクへ書き出す、ログ、回復処理

🎯 試験対策の合言葉
「"検査のタイミング" ときたら遅延制約の話！チェックポイント（障害回復）に引っかからない！」
