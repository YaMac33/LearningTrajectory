分散トランザクションと2相コミット（2PC）の仕組み

#データベース #トランザクション #2相コミット #基本情報 #応用情報

複数のデータベースが関わるシステムにおいて、最も重要な「全員で一斉に成功するか、全員で失敗するか（原子性）」を保証するための仕組み「2相コミット（2PC）」について解説します。

【よくある問題パターン】
「分散データベースシステムにおいて、最終的なコミットまたはロールバックを調停者（Coordinator）が決定する時点はいつか？」

➡ 答え：第2段階の開始時（全参加者からの可否応答が揃った時点）

1. なぜ2相コミットが必要なのか？

単一のデータベースなら「BEGIN → 処理 → COMMIT」で済みますが、銀行の振込のように「口座A（DB1）から減額し、口座B（DB2）へ加算する」といった複数のDBをまたぐ処理（分散トランザクション）では問題が起きます。

部分的な失敗による「データ不整合」の恐怖
🏦
DB 1 (口座A)
✔ 1万円 減額
振込処理 ➡
🏦
DB 2 (口座B)
❌ エラー発生!
(加算できない)

片方だけコミットされてしまうと、世界から「1万円」が消滅してしまいます。
これを防ぐため、全体の「原子性（Atomicity）」を保証する仕組みが必要です。

2. 2相コミット（Two-Phase Commit）の流れ

この問題を解決するために、全体を統括する「調停者（Coordinator）」と、各DBの「参加者（Participant）」に役割を分け、2つの段階（フェーズ）を踏んで確定させます。

第1段階（Prepare Phase：準備）

調停者が全員に「コミット可能か？」を問い合わせ、各DBは処理をして結果（OK/NG）を返します。

第2段階（Commit Phase：決定）

全員の「OK」が揃った時点で調停者が「コミット」を決定し、全員に指令を出します。（誰か1人でもNGなら全員にロールバックを指示）

3. チェックポイントとの「決定的な違い」

試験で最も狙われるのが、「チェックポイント」との混同です。対象とする範囲と目的が根本的に異なります。

🤝 2相コミット (2PC) 目的：全体で一致団結する（原子性）
  • 対象： 分散データベース（複数のDB）
  • 仕組み： 調停者が全員の合意をとってから確定させる
  • キーワード： 分散、調停者、可否確認、全員一致
💾 チェックポイント 目的：障害からの回復を速くする
  • 対象： 単一のデータベース
  • 仕組み： メモリの内容を定期的にディスクに書き出す「しおり」
  • キーワード： バッファ書き出し、回復時間の短縮、ログ

4. 2相コミットの弱点

安全性を高めるための2PCですが、通信や確認の手間が増えるため、以下のような弱点があります。

⚠ 調停者の故障による「ブロッキング」

第1段階で各DBが「OK（準備完了）」と返事をした直後に、調停者がダウンしてしまうと、各DBは「コミットしていいのか、ロールバックすべきなのか」が分からず、指示が来るまで永遠に処理を待機（ロック）し続けてしまうという致命的な弱点があります。

まとめ表

用語 役割・目的 対象環境
2相コミット (2PC) 全員が成功するか、全員が失敗するかを保証する（原子性の保証） 分散DB（複数）
チェックポイント 障害発生時の回復処理の時間を短縮する 単一DB

🎯 試験対策の合言葉
「"分散"・"調停者"・"可否の確認" ときたら『2相コミット』。回復を速くするのは『チェックポイント』！」
