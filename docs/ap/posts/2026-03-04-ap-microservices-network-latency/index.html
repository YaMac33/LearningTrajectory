マイクロサービスのネットワーク遅延と通信オーバーヘッド

#アーキテクチャ #マイクロサービス #ネットワーク #基本情報 #応用情報

1つの巨大なシステムを、機能ごとに小さなサービスに分割する「マイクロサービスアーキテクチャ」。開発やデプロイがしやすくなる一方で、「通信コスト」という新たな課題が発生します。

1. モノリシックとマイクロサービスの違い

システム内部の機能（認証、注文、在庫など）の繋がり方が根本的に異なります。

2. 1つの処理が引き起こす「通信の連鎖」

例えばユーザーが「商品を購入する」という1つのアクションを起こした時、バックエンドでは複数のサービス間でバケツリレーのように通信が発生します。

モノリシックなら一瞬で終わる処理が、ネットワークを何度も経由するため遅延が積み重なります。

3. オーバーヘッド（余計な処理コスト）の正体

同じサーバ内での「関数呼び出し」に比べ、ネットワーク越しの「HTTP/gRPC通信」は圧倒的に時間がかかります。

通信するたびに「接続の準備」や「データを送れる形に変換する処理」が発生します。これが通信オーバーヘッドです。

4. マイクロサービスにおける代表的な対策

この遅延問題に対処するため、マイクロサービスでは以下のような技術や設計パターンが組み合わされます。

🚪 API Gateway
クライアントからの複数のリクエストをまとめ、バックエンドとの通信回数を減らす。
⚡ キャッシュ
頻繁に変わらないデータ（商品情報など）を一時保存し、サービスへの問い合わせ自体を省略する。
📨 非同期通信 (キュー)
「注文完了」だけ先に返し、時間のかかる「メール送信」などは後から別のサービスにやらせる。
🕸️ サービスメッシュ
通信の制御や暗号化、再試行処理などをインフラ層で一括管理し、通信の信頼性を高める。

まとめ表

項目 モノリシック マイクロサービス
機能間の連携 メモリ内の関数呼び出し ネットワーク通信（HTTP/gRPC等）
処理スピード 非常に高速 遅延が発生しやすい
オーバーヘッド ほぼ無い TCP接続、シリアライズなどで増大
必要な対策 特になし（スケーリングが課題） API Gateway、非同期化、キャッシュ等
