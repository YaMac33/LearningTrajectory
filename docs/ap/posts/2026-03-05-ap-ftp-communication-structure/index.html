FTPの仕組み：制御とデータのチャネル分離

#ネットワーク #プロトコル #FTP #基本情報 #応用情報

インターネット黎明期から使われているファイル転送の標準プロトコル「FTP（File Transfer Protocol）」。他の多くのプロトコルとは異なる「2つの通信路を使う」というユニークな特徴と、試験で問われるポイントを解説します。

【よくある問題パターン】
「FTPにおいて、ユーザー認証やコマンドの送受信などの制御用コネクションと、実際のファイル転送用コネクションで使用されるTCPポート番号の組み合わせとして適切なものはどれか？」

➡ 答え：制御用に「21番」、データ転送用に「20番」

1. 制御チャネルとデータチャネルの「分離」

HTTPなどのプロトコルは1つの通信路（コネクション）で「リクエスト」も「データ」も送りますが、FTPは「指示を出すパイプ」と「荷物を運ぶパイプ」を明確に分けているのが最大の特徴です。

2. 接続の向き：アクティブとパッシブ

制御チャネル（TCP 21）は常に「クライアントからサーバ」へ接続しますが、データチャネル（TCP 20等）をどちらから繋ぎにいくかによって、2つのモードが存在します。

NATやFW環境下でも通信が通るため、現在はほぼこちらが使われます。

3. FTPの致命的な弱点と代替プロトコル

標準のFTPにはセキュリティ上の重大な欠陥があります。通信が一切暗号化されない（平文）ことです。

クライアント
━━━
ID: admin
PASS: secret123
━━━
FTPサーバ

パスワードやファイルの中身がそのままネットワーク上を流れるため、現在ではそのまま使われることは稀です。

安全にファイルを転送するために、現在は以下の暗号化プロトコルが利用されます。

  • FTPS： FTP通信を「SSL/TLS」で暗号化したもの。
  • SFTP： 「SSH」の仕組みを使って安全にファイル転送を行うもの。（※FTPとは全く別のプロトコルです）
