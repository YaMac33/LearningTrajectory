責任共有モデルと5つのセキュリティ

#AWS #セキュリティ #責任共有 #KMS #CloudHSM #Secrets Manager

AWSの責任共有モデルを理解し、KMSなど5サービスでの責任範囲を整理します。

「責任共有モデル (Shared Responsibility Model)」 は、AWSを利用する上での最も基本的なルールです。
セキュリティ系のサービス（KMS, CloudHSM, Secrets Manager, Parameter Store, ACM）は、このモデルにおいて「AWSがどこまでやってくれるか」がそれぞれ異なります。ここを整理すると、試験での正答率がグッと上がります。

1. 責任共有モデルの基本図解

大原則は以下の通りです。

責任の境界線
利用者の責任 IN the Cloud (クラウドの中身)
  • データの中身
  • 暗号化の設定
  • アクセス権限 (IAM)
責任分界点
AWSの責任 OF the Cloud (クラウドそのもの)
  • 物理データセンター
  • ハードウェア / ネットワーク
  • ハイパーバイザー

AWSは「土台」を守り、利用者はその上で扱う「データ」を守ります。

2. 各サービスにおける責任分担

以下の5つのサービスは、「利用者が果たすべき責任（データの保護など）」を助けるための道具です。
しかし、道具によって「手動でやる部分（自分）」と「全自動の部分（AWS）」の割合が違います。

① AWS KMS (Key Management Service)

  • AWSの責任: 鍵を作るハードウェア（HSM）の管理、物理セキュリティ。
  • 利用者の責任: 「どの鍵を使って暗号化するか」「誰に鍵を使わせるか（キーポリシー）」の設定。
  • 試験ポイント: 物理的な鍵管理はAWSに任せられます。

② AWS CloudHSM

  • AWSの責任: ハードウェアの電源が入っていること、物理的に壊れていないこと。
  • 利用者の責任: 鍵の生成、保管、バックアップ、OSのユーザー管理など、ほぼ全て。
  • 試験ポイント: 「オンプレミスの専用機」を借りているのに近いため、利用者責任が非常に重くなります。

③ AWS Secrets Manager

  • AWSの責任: サービスの稼働、パスワード保存時の暗号化。
  • 利用者の責任: 「どのパスワードを保存するか」「ローテーション（変更）の頻度設定」。
  • 試験ポイント: 秘密情報の「管理方法（どう回すか）」は利用者が決めます。

④ SSM Parameter Store

  • AWSの責任: パラメータを保存するデータベースの維持。
  • 利用者の責任: 「暗号化するかどうか（SecureStringを選ぶか）」、「誰に見せるか」。
  • 試験ポイント: 暗号化を忘れて平文で保存してしまった場合、それは利用者の責任です。

⑤ AWS Certificate Manager (ACM)

  • AWSの責任: 証明書の発行システム維持、更新期限の管理と自動更新。
  • 利用者の責任: 証明書をどのロードバランサー（ELB）やCloudFrontにセットするか。
  • 試験ポイント: 証明書の「更新作業」という面倒な運用責任を、AWSが肩代わりしてくれます。

3. サービス別「利用者責任の重さ」比較

試験で「運用負荷を減らしたい（＝責任をAWSに寄せたい）」のか、「統制を効かせたい（＝責任を自分で持ちたい）」のかで選び方が変わります。

利用者の負担（責任）レベル
CloudHSM
激重 (ほぼ自前)
Secrets Manager
中 (設定必要)
KMS
軽め (管理委任)
ACM
楽 (自動更新)

4. 試験での典型的な出題パターン

要件（ユーザーの要望） 正解のサービス 理由
「暗号鍵の管理責任をできるだけAWSに任せたい」 AWS KMS 物理管理や冗長化をAWSがやってくれるから。
「コンプライアンス要件により、鍵の管理を完全に自社で制御したい」 AWS CloudHSM AWSすら鍵にアクセスできない環境を作れるから。
「SSL証明書の更新作業を忘れないように自動化したい」 ACM 更新責任をAWSにオフロードできるから。

まとめ

  • 責任共有モデル: AWSは「クラウド自体の安全性」、利用者は「クラウド内の設定とデータ」に責任を持つ。
  • CloudHSM: 利用者の責任範囲が最も広い（自由だが大変）。
  • KMS / ACM: 利用者の責任範囲をAWSが肩代わりしてくれる（制約はあるが楽）。

試験では「誰が（AWSか利用者か）何をやる必要があるか？」という視点で問題文を読むと、正解が見えてきます。
