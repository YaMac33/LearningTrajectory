AWS CloudHSMの概要

#AWS #CloudHSM #暗号鍵 #セキュリティ #金融

CloudHSMは自分専用のハードウェア金庫で、暗号鍵を完全管理できるサービスです。

AWS CloudHSM（クラウド・エイチエスエム）は、一言で言うと「クラウド上に置く、あなた専用のハードウェア金庫」です。
AWSであっても中身を見ることができない、最強レベルのセキュリティ環境で暗号鍵を管理できます。
※HSM = Hardware Security Module（ハードウェアセキュリティモジュール）

1. KMSとの決定的な違い（メタファーで理解）

AWSには暗号鍵を管理するサービスとして「AWS KMS」がありますが、CloudHSMとは「誰が鍵を握っているか」が違います。

鍵の管理権限の違い
🏢
AWS KMS

「共用の鍵管理室」
AWSが管理する頑丈な部屋。
便利だが、管理者はAWS。

管理：AWS (共同)
🏦
CloudHSM

「自分専用の貸金庫」
ハード自体は銀行(AWS)にあるが、
開ける鍵は自分しか持っていない。

管理：自分 (専有)
  • AWS KMS: 「AWSが管理する安全なシステム」を使わせてもらうサービス。手軽で安価。
  • CloudHSM: 「AWSデータセンターにある物理的な機械（HSM）」を専有するサービス。AWS社員でも中身にはアクセスできません。

2. 責任分界点（どこまでやるか）

CloudHSMは「専用の機械」を借りるイメージに近いため、利用者（あなた）のやることが増えます。

CloudHSMの責任範囲
🔑 暗号鍵の生成・保管
誰にも見せずに自分でやる
👥 ユーザー管理
HSMへのログイン管理
⚙️ クライアントソフト
EC2へのインストール・設定
🏗️ 物理ハードウェア
故障交換・電源管理 (AWS)

3. どんな時に使うの？（利用シーン）

普通のWebサービスならKMSで十分です。CloudHSMが必要になるのは、以下のような「特殊で厳しい要件」がある場合のみです。

  • コンプライアンス規制: 金融業界や政府機関などで、「暗号鍵は自社で排他的に管理しなければならない（クラウド業者にも渡してはいけない）」というルールがある場合。
  • PKCS#11 などの標準規格: 既存のオンプレミスシステムが、特定の暗号化インターフェース（PKCS#11, Java JCE, Microsoft CNG）を使っており、それをそのままAWSに移行したい場合。

4. 試験対策比較表

CLF-C02やSAA試験では、「KMSでいいのか、CloudHSMが必要なのか」を判断させる問題が出ます。

項目 AWS KMS AWS CloudHSM
鍵の管理主体 AWS (と利用者の共有) 利用者のみ (完全管理)
テナント マルチテナント (共用) シングルテナント (専有)
利用難易度 簡単 (ポチッと有効化) 高い (専用クライアント設定が必要)
コスト 従量課金 (安い) 時間課金 (高い)
AWS連携 S3, EBS, RDS等と自動連携 一部対応だが、設定が複雑

まとめ

  • AWS CloudHSM: 鍵を自分で守るための「クラウド上の物理金庫」。
  • キーワード: 「専有ハードウェア」「キーの排他的管理」「コンプライアンス要件（金融・政府）」「PKCS#11」。

「AWSすら信用できない（してはいけない）レベルの厳しい要件」が出てきたら、CloudHSMの出番です！
