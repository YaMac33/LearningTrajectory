AWS EMRの概要と特徴

#Big Data #EMR #Hadoop #Spark #AWS

Amazon EMR (Elastic MapReduce) は、一言で言うと「Hadoop（ハドゥープ）やSpark（スパーク）といった『ビッグデータ処理フレームワーク』を、AWS上で超簡単に動かせるサービス」です。

AthenaやRedshiftが「SQL」でデータを扱うのに対し、EMRは「プログラミング（Python, Java, Scalaなど）」を使って、もっと複雑で大規模なデータ加工や機械学習を行うための基盤です。

試験では、「ビッグデータ」「Hadoop/Spark」「スポットインスタンスによるコスト削減」の3点が頻出です。

1. そもそも「Hadoop / Spark」とは？

これらは、1台のパソコンでは到底処理しきれないペタバイト級のデータを、数百台〜数千台のサーバーに分散させて、一気に計算する技術（オープンソース）です。

  • 課題: これらを自前（オンプレミス）で作ろうとすると、サーバーを何百台も買って、配線して、設定して…と、構築だけで数ヶ月かかります。
  • EMR: AWSコンソールでボタンを押すだけで、数分でこの「数百台のクラスター（計算センター）」が完成します。

2. EMRの仕組み：3種類のノード

EMRは、複数のEC2インスタンスを束ねて「クラスター」を作ります。この中にいるEC2（ノードと呼びます）には、3つの役割分担があります。

コスト削減のポイント:
タスクノードはデータを保存しないため、突然消えても問題ありません。
激安の「スポットインスタンス」を使うことで、処理費用を劇的に安くできます。

各ノードの詳細

  • マスターノード (Master Node): 必ず1台必要。全体の管理を行います。
  • コアノード (Core Node): 計算処理をしながら、データの一部（HDFS）を自分のディスクに保存・管理します。
  • タスクノード (Task Node): ★重要
    データの保存はせず、計算だけを手伝います。「マスターとコアは消えないオンデマンド、タスクは消えてもいいスポットインスタンス」という構成にするのが定石です。

3. 他のデータサービスとの違い（比較）

「大量のデータを処理する」サービスはたくさんあるので、迷いやすいポイントです。

サービス 得意なこと 手段 キーワード
Amazon Redshift データウェアハウス
構造化データの高速集計		 SQL DWH、ビジネス分析
Amazon Athena S3への直接クエリ
手軽なログ調査		 SQL サーバーレス、アドホック
AWS Glue ETLツール
データの抽出・変換・ロード		 Python / Scala サーバーレス、データカタログ
Amazon EMR ビッグデータ処理基盤
超大規模な計算・加工		 Hadoop / Spark クラスター、スポットインスタンス

Glue と EMR の違い:
Glueはサーバーレスで手軽ですが、中身はいじれません。EMRはサーバー構成（EC2）が見えるため、OSレベルの細かいチューニングや、既存のオンプレミスHadoop資産をそのまま移行したい時に選ばれます。

4. 具体的な利用シナリオ

  • ログ解析: Webサーバーのログ数億行を読み込み、機械学習を使って「不正アクセスの予兆」を見つける。
  • 金融シミュレーション: 株価データ過去30年分を使って、複雑なリスク分析の計算を行う。
  • オンプレミスからの移行: 自社データセンターでHadoopを使っていたが、ハードウェア更改が面倒なので、そのままAWS（EMR）に引っ越す。

まとめ

  • 役割: HadoopやSparkなどのビッグデータフレームワークをAWS上で簡単に動かすサービス。
  • 構成: EC2の集合体（クラスター）。マスター、コア、タスクの3種類。
  • コスト削減: 計算専用の「タスクノード」に「スポットインスタンス」を使うのが定石。
  • 使い分け: SQLで済むならRedshift/Athena。複雑なプログラミングが必要な大規模処理ならEMR。

これで、ビッグデータ処理の重鎮であるEMRについてもバッチリです。
