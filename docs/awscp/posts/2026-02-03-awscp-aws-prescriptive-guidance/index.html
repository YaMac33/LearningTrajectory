AWS Prescriptive Guidanceとは

#AWS #移行 #モダナイゼーション #ガイド #戦略

AWSの移行やモダナイゼーションを成功させるための実証済みの攻略法が集まったサイトです。

AWS Prescriptive Guidance (APG) は、AWS公式ドキュメントの中でも、特に「AWSのプロたちが実戦で培った『鉄板の攻略法』を集めたサイト」です。
通常のドキュメントが「このサービスの機能はこれです（機能説明）」という取扱説明書だとしたら、Prescriptive Guidanceは「こうやれば失敗せずに最短でゴールできます（戦略・手順）」という攻略本です。

一言で言うと、「AWSへの移行やモダナイゼーション（最新化）を成功させるための、『実証済みのレシピ集』」です。

1. なぜこれが必要なのか？

「機能の使い方（What）」がわかっても、「プロジェクトの進め方（How）」がわからないことはよくあります。
「EC2の使い方はわかった。でも、うちの会社の巨大なOracleデータベースを、ダウンタイムなしでAWSに移行するには、具体的にどういう手順で、どのツールを組み合わせてやればいいの？」
こうした具体的なシナリオに対する「正解」を提供するのがAPGです。

役割の違い
📖 通常ドキュメント 「取扱説明書」
機能の使い方、API仕様。
(例: 包丁の素材や握り方)
🍛 Prescriptive Guidance 「料理レシピ」
目的達成のための手順・戦略。
(例: 美味しいカレーの作り方)

2. 構成要素（3つの柱）

APGは主に以下の3つの要素で構成されています。

APGの構成要素
♟️
① Strategy (戦略)
ビジネスや組織をどう変えるかという「考え方」のガイド。
例: 「クラウド導入を社内でどう説得するか？」「7つのRの選び方」
🧭
② Guides (ガイド)
特定の目的を達成するための「計画書」。
例: 「メインフレーム移行ガイド」「VMware Cloud on AWS移行ガイド」
🛠️
③ Patterns (パターン) ★最重要
具体的な手順書（レシピ）。
アーキテクチャ図、必要なツール一覧、ステップ・バイ・ステップの手順、サンプルコードが含まれる。

3. 通常ドキュメントとの違い

試験や実務で迷わないための比較表です。

項目 通常のAWSドキュメント AWS Prescriptive Guidance
内容 機能 (What) の説明 方法 (How) と 戦略 (Why) の説明
視点 サービス単位 (EC2, S3など) 目的・課題単位 (移行, モダナイズなど)
対象 使い方を知りたい人 プロジェクトを成功させたい人
情報の出所 製品開発チーム AWSコンサルタント・エンジニアの実体験

4. 試験対策としてのポイント

CLF（クラウドプラクティショナー）やSAA（ソリューションアーキテクト）の試験で、もし以下のキーワードが出たら AWS Prescriptive Guidance を思い出してください。

  • 「移行 (Migration)」
  • 「モダナイゼーション (Modernization)」
  • 「実証済みのアプローチ (Proven approach)」
  • 「戦略とベストプラクティス (Strategies and best practices)」

特に、「オンプレミスからAWSへの移行プロジェクトを進めるにあたり、過去の成功事例に基づいた具体的な手順や戦略を知りたい。どこを見るべきか？」という問いの正解になります。

まとめ

  • AWS Prescriptive Guidance: AWS移行や活用を成功させるための「攻略本（レシピ集）」。
  • 中身: 戦略 (Strategy)、ガイド (Guides)、そして具体的なパターン (Patterns)。
  • 用途: 「どうやって移行すればいいの？」と迷った時に、先人の知恵を借りるために使う。
