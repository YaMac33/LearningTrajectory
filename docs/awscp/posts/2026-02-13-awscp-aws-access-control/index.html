AWSの部門別アクセス制御

#AWS #アクセス制御 #IAM #組織管理 #権限管理

500人規模の組織におけるAWSリソースの権限管理法を解説します。

Q. 「500人規模の組織で、各部門ごとにアクセス制御を設定する最適な方法は？」
A. 1つのアカウント内なら「IAMグループ」、完全に分離するなら「Organizations」です。

この問題は、「IAMグループ」「IAMロール」「Organizations」「個別管理」の使い分けを問う良問です。
それぞれの構成を図解して、なぜそれが正解（または不正解）なのかを整理しましょう。

1. 選択肢ごとの構成イメージ

管理手法の構造比較
◎ A. IAMグループ
🏢 AWSアカウント
👥 営業部グループ 権限A
👤 User 1
👤 User 2
👥 開発部グループ 権限B
👤 User 3

ユーザーをグループに入れて、グループに権限を貼る。管理が楽。

✕ B. IAMロール
🏢 AWSアカウント
👤 User 1
🎭 開発部ロール
⬆ 毎回AssumeRoleが必要

ロールは「一時的に被る帽子」。所属を表すものではない。

○ C. Organizations
🌏 Organizations
🏢 営業部アカウント
👤 User 1...
🏢 開発部アカウント
👤 User 3...

部門ごとに「アカウントごと」分ける。大規模ならアリだが構成が大掛かり。

✕ D. 個別設定
🏢 AWSアカウント
👤 User 1 権限A
👤 User 2 権限A
👤 User 3 権限B

一人一人にポリシーを貼る。500人いたら管理不能になる。

2. なぜ「IAMグループ」が正解なのか？

IAMのベストプラクティスは「権限は個人ではなくグループに付与する」です。

  • 効率性: 「営業部の権限を変更したい」と思った時、グループのポリシーを1回書き換えるだけで全員に反映されます。
  • 運用: 新人が入ったら「営業部グループ」に追加するだけ。異動したらグループを付け替えるだけです。

3. なぜ他はダメなのか？

  • B. IAMロール: ロールはEC2などの「サービス」や、「一時的な権限昇格」のために使うものです。「私は営業部です」という所属を表すために常時使うものではありません。
  • C. Organizations: これも正解になり得ますが、「アクセス制御を設定する方法」という文脈では、まずはIAMグループによる管理が基本です。アカウントを分けるのは「請求分離」や「強い隔離」が必要な場合です。
  • D. 個別設定: 500人のユーザー一人一人にポリシーをアタッチしていたら、誰が何の権限を持っているか把握できなくなり、セキュリティ事故の元です。

4. 規模別・使い分けの目安

規模・要件 推奨される方法 イメージ
小〜中規模
(1つのアカウント内)		 IAMグループ 「営業部」「開発部」というフォルダを作って整理する。
大規模・厳格
(請求も分けたい)		 AWS Organizations 「営業部株式会社」「開発部株式会社」のように会社（アカウント）ごと分ける。
一時利用 IAMロール 「今日だけ管理者権限を貸してあげる」ための入館証。

まとめ

「部門単位」の権限管理と言われたら、まずは「IAMグループ」を探してください。
（もし選択肢にIAMグループがなく、マルチアカウントの話をしていたら Organizations が正解になります）
