Provisioned Concurrencyの効果

#AWS #Lambda #Provisioned Concurrency #コールドスタート #レイテンシ

Provisioned ConcurrencyはLambdaの起動を事前に行い、低レイテンシを実現する機能です。

Q. 「LambdaはProvisioned Concurrencyにより、2桁のミリ秒単位で需要に対応できる」とはどういうことですか？
A. これは「事前にアイドリング状態を作っておくことで、寝起きの遅さ（コールドスタート）をなくす機能」の話です。

通常、Lambdaは「リクエストが来てから叩き起こされる」ため、最初の反応が遅れます。しかし、Provisioned Concurrency（プロビジョニングされた同時実行数）を設定すると、「常に準備運動済みのLambda」が待機することになり、いつ呼ばれても瞬時に反応できるようになります。

1. Lambdaの弱点「コールドスタート」

通常のLambdaは、しばらく使われていないと「停止状態」になります。その状態でリクエストが来ると、以下の手順を踏むため時間がかかります。

  1. 実行環境（コンテナ）を用意する
  2. プログラムコードをダウンロードする
  3. ランタイム（PythonやJavaなど）を起動する
  4. やっと処理を実行

この 1〜3 の準備時間を「コールドスタート」と呼び、数百ミリ秒〜数秒の遅延が発生します。

2. Provisioned Concurrency の仕組み

この機能を使うと、AWSが事前に 1〜3 の手順を済ませた状態でLambdaを待機させてくれます。

リクエスト発生時の挙動
通常
(オンデマンド)
⚡ リクエスト
準備 (遅い)
実行
Provisioned
(事前予約)
事前準備済み (待機)
⚡ リクエスト
即実行

準備時間がゼロになるため、いきなり処理が始まる（ホットスタート）。

3. サーバーレスなのに「待機」させる意味

「サーバーレス＝サーバー管理不要」ですが、Provisioned Concurrencyを設定するということは、実質的に「サーバーを常時起動させておく」のに近い状態になります。

裏側の待機イメージ
Lambda #1 Hot
Lambda #2 Hot
Lambda #3 Cold

設定した数（例: 2つ）だけは常にHot状態で待機。
それを超えた分（#3）は通常通りコールドスタートになる。

4. メリットとデメリット

項目 通常 (オンデマンド) Provisioned Concurrency
初回レイテンシ 遅い (コールドスタートあり) 爆速 (2桁ミリ秒で安定)
コスト 実行した分だけ (安い) 待機時間も課金される (高い)
向いている用途 バッチ処理、たまに動くボット Web API、決済処理、即時性が命の機能

まとめ

  • Provisioned Concurrency: Lambdaの「コールドスタート（寝起き）」をなくすための課金オプション。
  • 効果: いつリクエストが来ても、準備済みの環境が即座に応答するため、レイテンシが劇的に下がり安定する。
  • 注意点: 待機コストがかかるため、本当に速さが必要な重要な関数にだけ設定する。
