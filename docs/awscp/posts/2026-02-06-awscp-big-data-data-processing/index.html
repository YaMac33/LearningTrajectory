ビッグデータとデータ処理の関係

#ビッグデータ #データレイク #EMR #AWS #データ処理

ビッグデータはS3やEMRを駆使して、大量・多様・高速なデータを扱います。

Q. 「Big Data」というのは、S3（データレイク）のこと？それともEMR（処理）のこと？
A. どちらか一つではなく、「巨大な敵（課題）とその攻略チーム全体」のことです。

「ビッグデータ」という特定のAWSサービスがあるわけではありません。従来の技術では扱いきれないほど巨大で複雑なデータを扱う「分野」そのものを指します。
試験対策としては、「データレイク（S3）」も「加工（EMR）」も、すべてビッグデータを攻略するためのパーツであると理解するのが正解です。

1. ビッグデータの「3つのV」

ビッグデータには3つの特徴（3V）があり、AWSにはそれぞれに対応する「武器」があります。

ビッグデータの3要素とAWS
📦 Volume (量)
桁違いにデカい。
TB, PB, EB...
Amazon S3
🎨 Variety (多様性)
形式がバラバラ。
ログ, 画像, 音声
EMR / Glue
⚡ Velocity (速度)
猛スピードで来る。
リアルタイム
Kinesis
  • Volume（量）: 普通のハードディスクには入り切らない → S3（無限のストレージ）で受け止める。
  • Variety（多様性）: 構造が決まっていない（ログや画像） → EMR（加工工場）で整理する。
  • Velocity（速度）: 秒間数万件のデータが来る → Kinesis（土管）で流し込む。

2. 「データレイク」と「加工・処理」の関係

S3とEMRは、巨大なプロジェクトの中で役割分担をしています。これを「マグロ漁」に例えると非常に分かりやすくなります。

ビッグデータ活用の流れ
🐟 ビッグデータ (海・マグロ群)
➡
🧊 S3 データレイク
(冷蔵倉庫)
➡
🔪 EMR 加工・処理
(加工工場)
➡
🍣 Redshift DWH
(店舗陳列)
  • S3（データレイク）: 釣った大量のマグロ（生データ）を、とりあえず鮮度を保ったまま放り込んでおく「超巨大な冷蔵倉庫」。
  • EMR（加工・処理）: 倉庫からマグロを取り出し、解体してお刺身や缶詰にするための「加工工場」。
  • Redshift（DWH）: すぐにお客さんに出せる状態の切り身（集計済みデータ）をきれいに並べた「高級な陳列棚」。

3. なぜ両方必要なのか？

片方だけではビジネスになりません。

  • S3（倉庫）だけあっても: 「データは山ほどあるけど、ゴミの山で中身がわからない（食べる前のマグロ）」状態。
  • EMR（工場）だけあっても: 「加工する機械はあるけど、材料（データ）が手元にない」状態。

だからこそ、「S3に貯めてあるビッグデータを、EMRで処理する」というセット運用が基本になるのです。

4. 試験で問われやすいポイント

試験では「ビッグデータ」という用語定義よりも、「やりたいこと（要件）に対して、どのパーツを使うか？」が問われます。

やりたいこと (要件) キーワード 正解のサービス
構造化・非構造化を問わず、安価に大量保存したい データレイク Amazon S3
ペタバイト級のビッグデータを分散処理 (Hadoop/Spark) したい 加工・処理 Amazon EMR
ビッグデータを分析して可視化したい BI・ダッシュボード Amazon QuickSight

まとめ

「ビッグデータ」とは特定のサービス名ではなく、S3（貯める）、EMR（加工する）、Redshift（整理する）、QuickSight（見る）といったサービスを総動員して取り組む「一大プロジェクト」のことです！
