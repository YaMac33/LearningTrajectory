NATゲートウェイの一方通行の仕組み

  • NAT
  • ネットワーク
  • セキュリティ
  • データ通信
  • 仕組み

NATゲートウェイの「一方通行（中からは出れるが、外からは入れない）」という挙動は、セキュリティにおいて非常に重要です。一言で言えば、「自分から話しかけた相手の返事だけを通し、知らない人からの話しかけは無視する」という仕組みです。

これを実現している技術的な裏側である「変換テーブル（NAPTテーブル）」と、パケットの書き換え処理について、マンションの「コンシェルジュ（管理人）」に例えて図解します。

1. 登場人物とIPアドレス

仕組みを理解するために、以下のIPアドレスを設定します。

  • Aさん（Private EC2）： 10.0.0.5 （内線番号。外からは見えない）
  • NAT GW（Public）： 203.0.113.1 （代表電話番号。外から見える）
  • Google先生： 8.8.8.8 （インターネット上のサーバー）
  • ハッカー： 6.6.6.6 （攻撃者）

2. 「中→外」の通信（リクエスト）

Aさんが「Google先生、データをください」と依頼する時の流れです。

Aさん
10.0.0.5
手紙① From: 10.0.0.5 To: 8.8.8.8
➡ 依頼
NAT GW
203.0.113.1
📝 NATのメモ帳 (変換テーブル)
"10.0.0.5 が 8.8.8.8 と話してる"
(ポート番号 12345 を使用)
NAT GW
203.0.113.1
手紙② (書換) From: 203.0.113.1 To: 8.8.8.8
➡ 送信
Google
8.8.8.8

ポイント： NATゲートウェイは手紙の「送信元」を自分の住所に書き換え、同時に「誰の依頼か」をメモ帳（テーブル）に残します。

3. 「外→中」の戻り通信（返信）

Google先生からの返事が戻ってくる時の流れです。

Google
8.8.8.8
返信 ⬅
NAT GW
203.0.113.1
手紙③ From: 8.8.8.8 To: 203.0.113.1
📝 NATのメモ帳を確認
"お、これはさっきのAさん宛の返事だな"
→ 転送OK ✅
NAT GW
転送 ⬅
Aさん
10.0.0.5
手紙④ (戻し) From: 8.8.8.8 To: 10.0.0.5

ポイント： NATは届いた返信をメモ帳と照合し、Aさん宛だと分かったので宛先を書き戻して転送します。

4. なぜ攻撃（外→中）は防げるのか？

ハッカーがいきなりNATゲートウェイに攻撃パケットを送ってきた場合です。

ハッカー
6.6.6.6
攻撃手紙 From: 6.6.6.6 To: 203.0.113.1
➡ 攻撃
NAT GW
📝 NATのメモ帳を確認
"ん？ 6.6.6.6 なんて知らないぞ"
"内部の誰も話しかけてない"
→ 転送先不明 ❌
🗑️ 破棄

ポイント： NATはメモ帳に載っていない（内部からリクエストしていない）通信が来ても、「誰に転送していいか分からない」ため、その場で破棄します。AさんのIPアドレス（10.0.0.5）への転送ルールが存在しないため、物理的に届きません。

5. まとめ

  • 行き（Outbound）： NATが送信元を「自分」に書き換え、「誰の依頼か」をテーブルに記録する。
  • 帰り（Inbound）： テーブルに記録がある「返信」だけを、元の依頼者に転送する。
  • 攻撃（Unsolicited）： テーブルに記録がない通信は、転送先が不明なので破棄される。

これがNATゲートウェイが「マジックミラー」や「一方通行」と呼ばれる具体的な仕組み（NAPT：Network Address Port Translation）です。この仕組みのおかげで、プライベートサブネットは安全に守られています。
