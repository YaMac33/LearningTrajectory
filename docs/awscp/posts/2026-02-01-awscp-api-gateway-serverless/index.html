サーバーレスアーキテクチャのAPI Gateway

  • API Gateway
  • サーバーレス
  • Lambda
  • DynamoDB
  • 認証

Amazon API Gatewayは、サーバーレスアーキテクチャにおいて、「外の世界（ユーザー）」と「中の世界（Lambda）」を繋ぐ『窓口』です。EC2時代の「ELB（ロードバランサー）」に近い役割ですが、より「アプリケーション寄り（API専用）」の高機能な窓口として機能します。

1. なぜ API Gateway が必要なのか？

Lambdaはプログラムコードを実行する「凄腕の料理人（シェフ）」ですが、レストランの厨房の奥深くにいるため、インターネット（客席）から直接声をかけることはできません。客（ユーザー）が注文をするためには、注文を聞いて厨房に伝える「ウェイター」が必要です。それが API Gateway です。

2. 図解：サーバーレスの「ゴールデントリオ」

Webサービスを作る際、EC2などのサーバーを使わない「サーバーレス構成」の鉄板パターンがあります。

  • 昔の構成 (サーバー型): ELB ➔ EC2 ➔ RDS
  • 今の構成 (サーバーレス): API Gateway ➔ Lambda ➔ DynamoDB
「いいね！」ボタンを押した時の流れ
📱 ユーザー スマホアプリ
➡
🚪 API Gateway 受付・認証 ウェイター
➡
λ Lambda 処理実行 シェフ
➡
🗄️ DynamoDB データ保存 冷蔵庫

処理の流れ

  1. ユーザー: 「いいね！」リクエストを API Gateway のURLに送る。
  2. API Gateway (ウェイター): 「あ、常連さん（認証OK）ですね」「注文を受け付けました」。厨房の Lambda に「いいね処理をお願い！」と伝える。
  3. Lambda (シェフ): ロジックを実行し、DynamoDB の「いいね数」を +1 する。
  4. API Gateway: Lambdaからの完了報告を受け取り、ユーザーに「完了しました（200 OK）」と返す。

3. API Gateway の「3つのスゴ技」

API Gatewayはただの「通り道」ではありません。Lambdaを守るための重要な機能を備えています。

① スロットリング (流量制限)

Lambdaは際限なく同時実行できますが、その奥にあるDBには限界があります。「1秒間に1000回まで」と制限をかけることで、アクセスが殺到してもシステムがパンクするのを防ぎます。

② 認証・認可 (Security)

「誰でもこのAPIを叩ける」状態は危険です。AWSの認証サービス（Amazon Cognitoなど）と連携し、「ログイン済みのユーザーしか通さない」という門番の役割を果たします。

③ APIキーの使用量プラン

「無料プランの人は1日100回まで」「有料プランの人は無制限」といった、SaaSのような利用制限をかけることができます。

4. APIの種類（試験のミニ知識）

API Gatewayで作れるAPIにはいくつか種類があります。名前と特徴を押さえておきましょう。

  • REST API: 最も高機能。なんでもできる標準タイプ。
  • HTTP API: RESTより機能は少ないが、安くて速い。Lambdaを呼ぶだけならこれで十分なことが多い。
  • WebSocket API: リアルタイム通信（チャットアプリや株価ボード）用。サーバーとクライアントを繋ぎっぱなしにできる。

5. EC2時代の「ELB」との違い

「ELB（ALB）も振り分け役だったよね？ 何が違うの？」と迷うかもしれません。使い分けは以下の通りです。

機能 ELB (ALB) API Gateway
主な接続先 EC2 (常時起動サーバー) Lambda (サーバーレス)
得意なこと 大量のアクセスを、複数のサーバーに負荷分散する APIの管理（バージョン管理、認証、流量制限）
課金 起動時間課金 (1時間◯円) リクエスト数課金 (使った分だけ)

判断基準： EC2を使っているなら ELB、Lambdaを使っているなら API Gateway を選びます。

6. まとめ

  • 役割: サーバーレスアプリの「玄関口」。
  • 連携: API Gateway + Lambda + DynamoDB がサーバーレスの三種の神器。
  • 機能: ただ通すだけでなく、スロットリング（流量制限）や認証でバックエンドを守る。

これで、サーバーレス技術（Lambda, DynamoDB, API Gateway）まで完全に繋がりました。AWSのモダンなアプリケーション開発の基礎はバッチリです。
