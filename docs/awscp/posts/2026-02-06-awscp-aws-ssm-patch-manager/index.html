AWS SSM パッチマネージャーの全貌

#AWS #パッチ管理 #自動化 #セキュリティ #サーバー管理

SSM Patch ManagerでサーバーのOSアップデートを全自動管理。手動の限界を克服します。

AWS Systems Manager (SSM) Patch Manager は、一言で言うと「サーバーのOSアップデート（パッチ適用）を、全自動で一括管理するツール」です。
PCを使っていると「Windows Update」や「セキュリティ更新」が走って再起動を促されますよね？
あれを、1台ずつ手作業でやるのではなく、数百台・数千台のサーバーに対して、決められた時間に、決められたルールで勝手にやってくれる「司令塔」のようなサービスです。

1. なぜこれが必要なの？（手動更新の限界）

サーバーが100台あったとして、1台ずつログインして yum update や Windows Update を実行するのは不可能です。

管理スタイルの違い
😫 手動更新

1台ずつログイン...
「あれ？どこまでやったっけ？」

💻 💻 💻 💻
😎 SSM Patch Manager

画面でポチッとするだけ。
「全台一斉にやれ！」

📶 📶 📶 📶
  • 手間: 時間がかかりすぎる。
  • ミス: 「このサーバーだけ更新し忘れた」というセキュリティホールが生まれる。
  • 事故: 本番稼働中の昼間にうっかり再起動してしまい、サービスを止めてしまう。

2. Patch Manager の仕組み： 「3つの要素」

Patch Manager を使いこなすには、以下の3つの要素を組み合わせます。「ルール」「対象」「時間」です。

構成要素
📜

① パッチベースライン (Patch Baseline)

「更新のルールブック」です。
「重要度がCriticalなパッチは、公開後3日経ったら当てる」「このパッチ番号はバグがあるから除外する」といったルールを定義します。

🏷️

② パッチグループ (Patch Group)

「対象のチーム分け」です。
EC2につけたタグを使って、「開発環境グループ（すぐ当てる）」と「本番環境グループ（1週間様子見してから当てる）」のようにグループ化します。

⏰

③ メンテナンスウィンドウ (Maintenance Window)

「作業する時間」です。
勝手に再起動されると困るので、「毎週日曜日の 深夜2:00〜4:00 の間だけ作業してOK」という枠を指定します。

3. 試験に出る「Scan」と「Install」の違い

実行モードには2種類あります。目的によって使い分けます。

実行モード
🔍 Scan
チェックのみ
(現状把握・監査)
🔨 Install
適用 + 再起動
(メンテナンス)
  • Scan (スキャンのみ): パッチは当てません。「どのパッチが不足しているか？」だけをチェックしてレポートを作ります。
  • Scan and Install (スキャンしてインストール): 不足しているパッチを見つけて、実際にインストールし、必要なら再起動まで行います。

4. オンプレミスも管理できる！（最強のメリット）

SSM Patch Manager のすごいところは、AWS上のEC2だけでなく、オンプレミスにあるサーバーや、他社クラウドのVMも管理できることです。（※SSMエージェントさえ入っていればOK）

これにより、「会社の全サーバーのセキュリティ状態」をAWSの画面一つで統一管理できるようになります。

まとめ

  • SSM Patch Manager: 大量のサーバーのOS更新を自動化するツール。
  • 3要素: ベースライン（ルール）、グループ（対象）、ウィンドウ（時間）。
  • メリット: ログイン不要で一括管理。オンプレミスもOK。

「セキュリティ担当者が、全サーバーに号令をかけて一斉にワクチン（パッチ）を打たせる司令室」だとイメージしてください！
