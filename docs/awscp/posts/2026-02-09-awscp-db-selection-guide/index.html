DB選定ガイド

#データベース #システム設計 #エンタープライズ #AWS #Redis

ECサイトやゲーム等のシステム構成とDB選定理由を図解。使用用途別にまとめました。

AWSには多数のデータベースサービスがありますが、「どんなシステムで」「なぜそのサービスを選ぶのか」を理解するのが設計の第一歩です。
具体的なシステム構成例を見ながら、RDS、DynamoDB、ElastiCache、MemoryDBの役割分担を整理しましょう。

1. 全体マップ（まずは俯瞰）

各データベースの「得意なポジション」は以下の通りです。

AWSデータベースのポジション
RDS 永続・リレーショナル
(契約・決済)
ElastiCache キャッシュ専用
(消えてもOK)
MemoryDB 超高速・高耐久
(消えたら困るRedis)
DynamoDB NoSQL・永続
(大量データ)

2. システム例①：ECサイト（全部入り）

ECサイトは、さまざまな種類のデータを扱うため、複数のDBを適材適所で組み合わせる典型例です。

ECサイトのアーキテクチャ
API / Webサーバー
⬇
MemoryDB カート・セッション
ElastiCache 商品一覧キャッシュ
RDS 注文・決済
⬇
DynamoDB 商品マスタ / 在庫 / 履歴

選定理由

  • RDS: お金に関わる「注文・決済」は、整合性（トランザクション）が最重要なのでRDB一択。
  • DynamoDB: 商品データや在庫はアクセス数が桁違いに増えるため、スケーラビリティの高いNoSQLが最適。
  • ElastiCache: 商品一覧の表示は速くしたいが、元データはDBにあるので、キャッシュが消えても再取得すればOK。
  • MemoryDB: 「買い物かご」の中身は超高速に読み書きしたいが、サーバー障害で中身が消えると「カゴ落ち（機会損失）」になるため、耐久性のあるMemoryDBを採用。

3. システム例②：ゲーム（ランキング＋履歴）

リアルタイム性が命のゲームでは、Redisの構造が役立ちます。

ゲーム・ランキング構成
MemoryDB リアルタイム順位
⬇ 定期確定
DynamoDB スコア履歴
RDS 課金・購入

選定理由

  • MemoryDB: Redisの「Sorted Set」機能を使うと、ランキング計算が爆速です。順位データが飛ぶとクレームになるのでElastiCacheではなくMemoryDB。
  • DynamoDB: 過去の対戦履歴など、膨大なログデータを保存するのに最適。

4. システム例③：SaaS / 業務システム

BtoBの業務システムでは、安定性とコストバランスが重視されます。

業務システム構成
ElastiCache 画面キャッシュ
⬇
RDS 顧客・契約・請求

選定理由

  • ElastiCacheで十分: 業務システムのキャッシュは「消えてもDBから取り直せばいい」ものが多いため、高価なMemoryDBは不要です。
  • RDS: 複雑な結合（JOIN）や集計が必要な業務データにはRDBが必須。
