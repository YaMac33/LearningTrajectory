AWS Systems Managerの基本機能

#AWS #SSM #運用自動化 #Session Manager #Run Command #Patch Manager

AWS Systems Managerでサーバー運用の自動化を実現。SSH不要の操作で安全かつ効率的。

AWS Systems Manager (SSM) は、AWS上の大量のサーバー（EC2）やオンプレミスのサーバーを、「一括で管理・操作するための統合コックピット」です。
一言で言うと、「もうサーバーにログインしなくていい。SSH鍵もいらない。全部この画面から遠隔操作できる」という革命的なサービスです。

試験では、以下の3点が超頻出キーワードとなります。

  • 運用自動化
  • パッチ適用
  • SSH不要（セッションマネージャー）

1. 最大の目玉機能：Session Manager（セッションマネージャー）

これまでサーバー（EC2）に入ってコマンドを打つには、以下の準備が必要でした。これらはすべてセキュリティリスクや管理の手間につながります。

  • 22番ポート（SSH）を開ける（セキュリティリスク）
  • SSH鍵（pemファイル）を管理する（紛失リスク）
  • 踏み台サーバー（Bastion）を経由する（面倒）

SSM Session Manager を使うと、これらが全部不要になります。ブラウザ上のクリックだけで、安全に黒い画面（シェル）が開きます。

主なメリット

  • ポート開放不要: インバウンドポートを全部閉じても入れます（アウトバウンド443があればOK）。
  • 鍵不要: IAM権限で認証するので、鍵ファイル（pem）が要りません。
  • ログ記録: 「誰がどんなコマンドを打ったか」が全部記録されます（S3やCloudWatch Logsへ）。

2. その他の主要機能（ツールボックス）

SSMは多機能ですが、試験に出る重要機能は以下の3つです。

① Run Command（ランコマンド）

「一斉送信」機能です。
「サーバー100台全部で、Apacheを再起動したい」という時、1台ずつログインするのは現実的ではありません。Run Commandを使えば、ターゲットを指定してボタンを押すだけで、100台同時にコマンドを実行できます。

② Patch Manager（パッチマネージャー）

「全自動アップデート」機能です。
「毎週土曜日の夜中2時に、Windows Updateを適用して再起動する」といった作業を自動化します。「メンテナンスウィンドウ」という機能と組み合わせて、業務に影響がない時間に実行するのが一般的です。

③ Parameter Store（パラメータストア）

設定値やパスワードを保存する機能です。Secrets Managerの簡易版として、より低コストで使用されます。

SSM許可ポリシー
  • SSM エージェント (SSM Agent)
    • EC2の中に、SSMからの命令を受け取る「駐在員ソフト（エージェント）」がインストールされ、起動している必要があります。
    • ※Amazon Linux 2 / 2023 など、最近の主要OSには最初から入っています。
  • IAM ロール
    • EC2に、「SSMと通信してもいいよ」という権限（IAMロール） がアタッチされている必要があります。
    • これがないと、エージェントが入っていてもSSMは操作できません（代表的なポリシー: AmazonSSMManagedInstanceCore）。

4. OpsWorks や SSH との比較

「サーバーを管理する」手段としての使い分けは以下の通りです。

手段 特徴 今の推奨
SSH (手動) 1台ずつ鍵を使ってログイン。ポート開放が必要。 非推奨 (セキュリティリスクが高い)
OpsWorks Chef/Puppetを使う。複雑な設定管理向け。 特殊な場合のみ (既存資産がある時など)
SSM (Session Manager) ポート開放不要、鍵不要。ブラウザから操作。 ★基本はこれ (最も安全で楽)
SSM (Run Command) 複数台への一括実行。 ★基本はこれ (運用の自動化)

まとめ

  • 役割: EC2やオンプレサーバーの運用管理を自動化・統合するサービス。
  • Session Manager: SSH不要、ポート開放不要、鍵不要でサーバーにログインできる最強機能。
  • Run Command: 複数サーバーへのコマンド一括実行。
  • Patch Manager: OSのセキュリティパッチ適用を自動化。
  • 必須条件: サーバーに「SSMエージェント」と「IAMロール」が必要。
