AP型データベースの特徴と例

#AP型 #データベース #CAP定理 #可用性 #整合性

「AP型データベース」とは、「データの厳密な正しさよりも、『システムが絶対に止まらないこと』を最優先したデータベース」のことです。

これは「CAP定理（キャップていり）」という、ITエンジニアにとっての「ジャンケンのルール」のような理論に基づいた分類です。詳しく解説します。

1. 前提知識：CAP定理（3つの願い）

分散システム（クラウド上のDBなど）には、以下の3つの理想があります。しかし、「この3つを同時にすべて満たすことは不可能で、最大で2つしか選べない」という絶対的なルールがあります。これがCAP定理です。

クラウドでは通信断絶のリスクがあるため、P（分断耐性）は必須です。
実質的に、「C（正しさ）をとるか、A（止まらなさ）をとるか」の二択になります。

  • C (Consistency：整合性): いつ、誰が見ても、必ず「最新の正しいデータ」が見えること。（例：銀行の残高）
  • A (Availability：可用性): システムがダウンせず、「いつでも応答」すること。（例：SNSの「いいね」ボタン）
  • P (Partition Tolerance：分断耐性): 通信障害が起きてもシステムが動き続けること。（※クラウドでは必須）

2. AP型データベースの特徴

「A（可用性）」と「P（分断耐性）」を選んだデータベースです。

  • スローガン: 「多少データが古くてもいいから、絶対にお客さんを待たせるな！」
  • 犠牲にしたもの: C（即時の整合性）
    • 「書き込んだ直後に読み込むと、まだ古いデータのままかもしれない」という状態を許容します。
    • これを専門用語で「結果整合性（Eventual Consistency）」と呼びます。「最終的に（数秒後には）辻褄が合うからOK」という考え方です。

4. AWSにおける AP型

AWSのサービスで言うと、Amazon DynamoDB が代表的なAP型データベースです。

  • DynamoDBのすごいところ:
    • 基本は「AP型（結果整合性）」で爆速で動きます。
    • しかし、設定オプションで「強力な整合性（Strongly Consistent Read）」を選ぶことで、無理やり「CP型」のような振る舞いをさせることも可能です（その代わり速度は落ち、コストは上がります）。

まとめ

  • AP型: Availability（可用性）重視。「多少データがズレてもいいから、絶対にサービスを止めるな」という設計。
  • CP型: Consistency（整合性）重視。「止めてでもいいから、絶対にデータの矛盾を起こすな」という設計。
  • CAP定理: 「正しさ(C)」と「止まらなさ(A)」の両立は無理。どっちか選べというルール。

「AP型」と聞いたら、「DynamoDBのような、結果整合性を受け入れてスピードと稼働率を極めたデータベース」と思い出せればOKです！
