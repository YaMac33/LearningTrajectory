ElastiCacheの高速データ処理

#ElastiCache #Redis #Memcached #データベース #AWS #キャッシュ

Amazon ElastiCache（エラスティキャッシュ）は、「ミリ秒」を「マイクロ秒」にするための、爆速のデータ一時保存サービスです。

一言で言うと、「データベース（RDS）の手前に置く、超高速な『カンニングペーパー（キャッシュ）』」です。試験では、「データベースの負荷軽減」「読み込み速度の高速化」「RedisとMemcachedの違い」が頻出です。

1. なぜ ElastiCache が必要なのか？

通常のデータベース（RDSなど）は、データが「ディスク（HDD/SSD）」に保存されています。これは安全ですが、物理的な読み書きが発生するため、スピードには限界があります。

※ 1ミリ秒 = 1,000マイクロ秒。桁違いの速さです。

仕組み：Lazy Loading（遅延読み込み）

  1. アプリはまず、ElastiCacheを見に行きます。「答え（キャッシュ）ある？」
  2. あれば、即座に返します（キャッシュヒット）。爆速です。
  3. なければ、RDSに問い合わせてデータを取りに行き、その結果をElastiCacheに書き込んでおきます（キャッシュミス）。
  4. 次回からはElastiCacheが答えてくれます。

2. 試験のハイライト：「Redis」vs「Memcached」

ElastiCacheを使う時、「Redis（レディス）」と「Memcached（メムキャッシュド）」という2つのエンジンからどちらかを選ばなければなりません。この2つの違いが試験で最も問われます。

合言葉 「高機能」「バックアップ」「ランキング」 「シンプル」「マルチスレッド」

★試験対策:
迷ったら Redis を選ぶのが基本です。現在のAWSでは、ほとんどのケースで高機能なRedisが推奨されます。「Memcached」が正解になるのは、「極めてシンプルな構造で、マルチスレッド性能が必要」と明記された時くらいです。

3. 具体的なユースケース

ElastiCacheは「データベースの代わり」ではなく、「データベースを助ける」ために使われます。

① データベースの負荷軽減 (Database Caching)

「商品カタログ」や「トップページのニュース」など、「みんなが同じものを見るけど、滅多に変わらないデータ」をキャッシュします。RDSへのアクセスが減り、RDSのスペックを下げてコストダウンできます。

② セッション管理 (Session Store)

ユーザーの「ログイン情報」や「ショッピングカートの中身」を保存します。Webサーバーが複数台あっても、ElastiCacheに情報を預けておけば、どのサーバーに繋がっても「あ、ログイン中ですね」と認識できます。

③ リアルタイムランキング (Gaming Leaderboards)

Redisの「ソート済みセット（Sorted Sets）」という機能を使うと、ゲームのスコアランキングなどをリアルタイムに集計・表示できます。RDSでこれをやるとサーバーが死にます。

4. DynamoDB Accelerator (DAX) との違い

「キャッシュ」といえばもう一つ、DAX というサービスがありますが、これもしょっちゅう比較されます。

  • ElastiCache: RDS やその他のDB（汎用的）の手前に置くキャッシュ。
  • DAX: DynamoDB 専用 のキャッシュ。

試験で「DynamoDBの読み込みを高速化したい」と言われたら、ElastiCacheではなく DAX が正解です。

まとめ

  • 役割: インメモリ（メモリ上）で動く超高速キャッシュサービス。
  • メリット: データの読み込みをマイクロ秒単位にする。DBの負荷を下げる。
  • Redis: 高機能。ランキング、バックアップ、マルチAZが必要ならこっち（基本はこれ）。
  • Memcached: シンプル。単純なキャッシュならこっち。

これで、データベースを高速化する重要パーツであるElastiCacheも攻略完了です。
