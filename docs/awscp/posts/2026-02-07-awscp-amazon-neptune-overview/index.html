Amazon Neptuneの概要と特徴

#AWS #Neptune #データベース #グラフDB #フルマネージド

Amazon Neptuneは、データ同士の「関係性」を高速に扱うためのグラフ型データベースです。

Amazon Neptune（ネプチューン） は、一言で言うと「人やモノ同士の『関係（つながり）』を高速に扱うためのグラフ型データベース」です。
RDS（リレーショナルデータベース）や DynamoDB（NoSQL）とは「発想」が根本的に異なります。ここを理解すると、試験での正答率が一気に上がります。

1. 「グラフデータベース」って何？

普通のデータベース（RDS）とグラフデータベース（Neptune）の最大の違いは、「何が主役か」です。

データ構造の違い

  • 普通のDB（RDSなど）: 「表（テーブル）」でデータを管理します。AさんとBさんの関係を知るには、テーブル同士をくっつける計算（JOIN）が必要です。
  • グラフDB（Neptune）: 「点（ノード）」と「線（エッジ）」で管理します。「Aさん →（友達）→ Bさん」という「つながり」そのものをデータとして持っているのが特徴です。

Neptuneは「関係をたどる処理（トラバーサル）」が爆速です。

典型的なユースケース

  • SNS: 「友達の友達を探す」「知り合いかも？を表示する」
  • レコメンドエンジン: 「この商品を買った人が、次に買った商品」を提案する。
  • 不正検知: 「怪しいアカウントと繋がっている別のアカウント」を芋づる式に見つける。

3. 対応モデルと構成

Neptuneはフルマネージドサービスなので、バックアップや冗長化（マルチAZ）はAWSがやってくれます。RDSに近い安心感があります。
また、以下の2つのグラフモデルに対応しています（試験では「2種類ある」ことだけ覚えればOK）。

  • プロパティグラフ (Gremlin): ノードやエッジに自由に属性を持たせる。SNS向き。
  • RDF (SPARQL): 主語・述語・目的語で厳密に定義する。学術・知識グラフ向き。

4. RDS / DynamoDB との使い分け（試験対策）

観点 RDS DynamoDB Amazon Neptune
データ構造 表 (Table) Key-Value グラフ (Graph)
得意な処理 集計・JOIN 単純な高速検索 関係探索・深掘り

試験での「Neptune即答ワード」

問題文に以下の単語が出たら、Neptuneを疑ってください。

  • 「関係性 (Relationships)」
  • 「グラフ (Graph)」
  • 「つながりを辿る」
  • 「レコメンデーション (Recommendation)」
  • 「ソーシャルネットワーク」

まとめ

  • Amazon Neptune: データ同士の「関係性」をグラフ構造で保持し、高速に探索できるフルマネージドDB。
  • 使い所: 「友達の友達」のような「多段階のつながり」を調べる時に最強。

「複雑なデータ構造だからNeptune」は間違いです。「関係をたどる必要があるからNeptune」が正解です！
