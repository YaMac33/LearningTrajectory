Amazon EventBridgeの基礎

#AWS #EventBridge #CloudWatch #Lambda #自動化

Amazon EventBridge（イベントブリッジ）は、一言で言うと「AWSのサービス同士や、外部のSaaSアプリを『イベント』をきっかけにつなぐ、万能の仲介役（ハブ）」です。

以前は「Amazon CloudWatch Events」と呼ばれていましたが、機能が強化されて名前が変わりました。もし古い参考書に「CloudWatch Events」とあったら、それは現在のEventBridgeのことだと思ってください。

1. なぜ名前が変わったの？（最大の違い）

以前の「CloudWatch Events」は、主に「AWSの中のこと（例：EC2が落ちた）」しか扱えませんでした。

しかし、「Salesforceでお客さんが登録されたら」とか「Zendeskでチケットが作られたら」といった、「AWSの外（SaaS）」の情報も扱えるようにパワーアップしました。そこで、「AWSの中だけじゃないぞ、いろんなサービスの架け橋（Bridge）になるぞ」という意味で EventBridge という名前に変わりました。

2. EventBridge の仕組み

EventBridgeは「イベントバス」という通り道を持っています。ここに情報が集まり、仕分けされて、次の処理へ渡されます。

  • ソース（発生源）: 「EC2が停止した」「午後3時になった」「SaaSで注文が入った」など。「何か起きたよー！」と叫ぶ人たちです。
  • イベントバス（受け皿）: 叫ばれた内容（イベント）を受け止めるパイプラインです。
  • ルール（仕分け人）: 「『EC2停止』ならAへ」「『午後3時』ならBへ」と、内容を見て行き先を決めます。
  • ターゲット（宛先）: 実際に処理を行う Lambda や SNS などです。

3. 3大メリット：何が便利なの？

① 外部SaaSとの連携（これが最強）

今まで「Salesforceにデータが入ったらAWSで処理する」には複雑なプログラムが必要でしたが、EventBridgeなら設定だけで連携できます。

② システムの「疎結合（Decoupling）」

AシステムがBシステムを直接呼び出すと、片方の変更がもう片方に影響します（密結合）。間にEventBridgeを挟むことで、お互いを知らなくて済み、壊れにくいシステムになります。

③ 高機能なスケジューラー

「毎朝9時にバックアップ」といった定期実行も可能です。最近追加された「EventBridge Scheduler」を使えば、タイムゾーン指定や祝日の考慮など、より細かい管理が可能です。

4. CloudWatch / Lambda との違い（混乱ポイント）

AWSで自動化を作る時の「ゴールデントリオ」です。役割が混同しやすいため、「ビルの防災システム」に例えて整理しましょう。

まとめ

  • 正体: CloudWatch Events の超・進化版。
  • 役割: サービス間に入って「何か起きたら（イベント）、あっちを動かす」という仲介役。
  • 強み: AWSだけでなく、SaaS（Salesforceなど）とも連携できる。
  • 試験対策: 「サードパーティ」「SaaS連携」「カスタムイベントバス」という単語が出たら EventBridge。

「目（CloudWatch）で見て、神経（EventBridge）が伝えて、手（Lambda）が動く」。この流れを覚えておけば完璧です！
