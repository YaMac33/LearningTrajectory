CloudFormationによるリソース構築

#CloudFormation #AWS #EC2 #S3 #VPC #IaC

EC2、S3、VPCといった複数のAWSリソースを、CloudFormationを使って「ひとまとめ」に自動構築する応用例を紹介します。

CloudFormationの最大のメリットは、個々のサービスをバラバラに作るのではなく、「システム全体を1つのテンプレート（設計図）として管理できる」点にあります。

1. テンプレートの全体像（YAML形式）

以下は、VPC（ネットワーク）、EC2（サーバー）、S3（ストレージ）を一度に作成するシンプルなテンプレートの例です。 

AWSTemplateFormatVersion: '2010-09-09'
Description: 簡単なVPC + EC2 + S3構築例

Resources:
  # ① VPCを作成
  MyVPC:
    Type: 'AWS::EC2::VPC'
    Properties:
      CidrBlock: 10.0.0.0/16
      Tags:
        - Key: Name
          Value: MyVPC

  # ② EC2インスタンスを作成
  MyEC2Instance:
    Type: 'AWS::EC2::Instance'
    Properties:
      InstanceType: t2.micro
      ImageId: ami-0abcdef1234567890
      SubnetId: !Ref MyVPC  # ★ここでVPCを参照！

  # ③ S3バケットを作成
  MyS3Bucket:
    Type: 'AWS::S3::Bucket'
    Properties:
      BucketName: my-app-bucket

2. 図解：コードと実体の関係

コードに書いた内容が、実際にどのような構成としてデプロイされるかを視覚化しました。 ポイントは !Ref による「リソース間の紐付け」です。

テンプレートから生成される環境
Template.yaml
Resources: MyVPC: AWS::EC2::VPC ... MyEC2: AWS::EC2::Instance SubnetId: !Ref MyVPC ... MyS3: AWS::S3::Bucket
➡
AWS Stack
MyVPC (Network)
MyEC2Instance
MyS3Bucket

リソースの解説

論理ID (コード上の名前) AWSリソース 説明
MyVPC AWS::EC2::VPC 独自のネットワーク空間を作成します。
MyEC2Instance AWS::EC2::Instance 仮想サーバーです。!Ref MyVPC を使うことで、「さっき作ったVPCの中に配置する」という依存関係を定義しています。
MyS3Bucket AWS::S3::Bucket 独立したデータ保存領域です。EC2とは直接依存しませんが、同じスタック内で管理されます。

3. 実務でのメリット

このように1つのファイルにまとめることで、以下のような運用が可能になります。

  • 環境の完全コピー: 全く同じ構成の「開発環境」や「ステージング環境」を、テンプレートを再利用するだけで一瞬で作れます。
  • 依存関係の解決: 「VPCを作ってからじゃないとEC2は作れない」といった作成順序を、CloudFormationが自動で判断してくれます。
  • 一括削除: スタックを削除すれば、EC2もS3もVPCも、依存関係を考慮しながら綺麗に消してくれます。

4. 試験対策ポイント (CLF-C02)

  • Resourcesセクション: ここに作りたいリソース（EC2やS3）を記述します。必須項目です。
  • まとめて構築: CloudFormationは「複数の関連するリソース」をスタックとして一括管理するサービスです。
  • 自動化: AWS CLIで aws cloudformation deploy コマンドを叩けば、画面操作なしでインフラが完成します。

まとめ

「CloudFormation = 複数リソースをまとめて作れる設計図」

この設計図さえあれば、何度でも同じ環境を再現できるのがIaC（Infrastructure as Code）の強みです。
