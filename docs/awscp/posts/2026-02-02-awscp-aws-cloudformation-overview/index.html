AWS CloudFormationの概要

#AWS #CloudFormation #IaC #自動化 #テンプレート

AWS CloudFormation（クラウドフォーメーション）は、AWSのインフラ構築を「完全自動化」するサービスです。

一言で言うと、「AWSの『設計図（コード）』を書けば、ボタン一つでその通りの環境を全自動で作ってくれるサービス」です。これを専門用語で 「Infrastructure as Code (IaC)」 と呼びます。試験で最も重要なキーワードの一つです。

1. なぜ CloudFormation が必要なのか？

もし、あなたが「Webサーバー(EC2)と、データベース(RDS)と、ネットワーク(VPC)のセット」を、本番環境、開発環境、テスト環境の3つ分用意してくれと言われたらどうしますか？

2. 仕組み：3つの構成要素

CloudFormationを理解するには、以下の3つの段階を覚えるだけでOKです。

① テンプレート (Template) = 「設計図」

「EC2を1台、S3を1つ作ってください」という指示を書いたテキストファイルです。

  • 形式: JSON または YAML（ヤムル） という形式で書かれます。

② スタック (Stack) = 「完成した建物」

テンプレートをCloudFormationに読み込ませると、実際に作られるAWSリソースの「ひとまとまり」のことです。

  • 重要: スタックを削除すると、その中に含まれるEC2やRDSも一括で削除されます。お片付けが超簡単です。

③ CloudFormation エンジン = 「建設ロボット」

設計図（テンプレート）を読み取って、実際にAWS APIを叩いてリソースを作ってくれる本体です。

3. 試験に出る「重要機能」

CloudFormationの問題が出たら、以下のキーワードを探してください。

① 変更セット (Change Sets)

スタックを更新する時の「確認画面」です。

  • 「テンプレートを修正したけど、これを適用したら何が変わるの？」を事前にチェックできます。
  • 例: 「これを適用すると、本番DBが削除されます」といった恐ろしい事故を未然に防げます。

② ドリフト検出 (Drift Detection)

「設計図と、現実のズレ」を検知する機能です。

  • シナリオ: 誰かが勝手に手動でEC2の設定を変更してしまった（設計図には書いてないのに！）。
  • 機能: CloudFormationは「あれ？設計図と違いますよ」と教えてくれます。これを「ドリフト（漂流）」と呼びます。

③ デザイナー (Designer)

コードを書くのが苦手な人のために、GUI（ドラッグ＆ドロップ）で図を描けば、自動的にJSON/YAMLコードを生成してくれるツールです。

4. AWS Elastic Beanstalk との違い

「自動で環境を作る」という意味でよく混同されますが、守備範囲が違います。

特徴 CloudFormation Elastic Beanstalk
役割 インフラの自動化 (IaC) アプリのデプロイ・管理 (PaaS)
自由度 無限大
(AWSのほぼ全サービスを操作可能)		 制限あり
(Webアプリの定番構成のみ)
対象 ネットワーク設計や細かいセキュリティ設定までこだわりたい人 インフラのことは忘れて、アプリコードを動かしたいだけの人
たとえ 「注文住宅」
(柱一本から指定できる)		 「建売住宅」
(標準的な家を即購入)

試験対策:
・「アプリを手軽にデプロイしたい」 → Elastic Beanstalk
・「インフラ構成をコードで管理したい」「再現性を高めたい」 → CloudFormation

まとめ

  • IaC (Infrastructure as Code): インフラをコード（テキスト）で管理する。
  • テンプレート: JSONかYAMLで書かれた設計図。
  • スタック: テンプレートから作られたリソースの集合体。一括削除が可能。
  • メリット: 誰が作っても全く同じ環境ができる（再現性）、Gitなどでバージョン管理ができる。

これで、自動化ツールの王様であるCloudFormationについてもバッチリです。
