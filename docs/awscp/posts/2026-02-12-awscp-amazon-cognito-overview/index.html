Amazon Cognitoの概要と利用法

#Cognito #AWS #認証 #アプリ #セキュリティ

自作アプリにログイン機能を簡単に実装し、AWSと安全に連携させる「Cognito」の仕組みを解説します。

「自作アプリのユーザーにアカウントを作らせて、マイページを表示させたい」
これを自前で実装しようとすると、パスワードの暗号化やセッション管理など、セキュリティの壁にぶち当たります。これを丸ごと引き受けてくれるのが Amazon Cognito です。

「自前で認証サーバーを作らなくていい」のが最大のメリットですが、Cognitoには「ログイン機能」と「AWSとの連携」という2つの顔があるため、少し複雑に見えがちです。

1. Cognitoの2つの機能（プール）

Cognitoの中には、役割が全く違う2つの「プール」が存在します。ここを切り分けるのが理解の第一歩です。

ユーザープールとIDプールの役割
🏢 ユーザープール アプリの「受付」
  • サインアップ・ログインの処理
  • パスワードの保存・管理
  • MFA（多要素認証）
  • ログイン成功時に JWTトークン を発行
🎫 IDプール AWSの「入館証発行所」
  • ユーザープールでログインした人を確認
  • STSと連携し 一時クレデンシャル を発行
  • 「この人はS3の自分のフォルダだけ見てヨシ」などの権限（IAMロール）を渡す

「ログインさせるだけ」ならユーザープールのみでOK。
「アプリから直接S3に画像をアップロードさせたい」場合はIDプールも使います。

2. 実践：マイページはどう動いているのか？

「ログインしたユーザーが、自分の履歴だけをDynamoDBから取得する」という定番のフローを見てみましょう。

安全なデータアクセスの流れ
📱
1. ログイン ユーザーがID/Passを入力
⬇
🏢
2. 認証OK！トークン発行 Cognito (ユーザープール) が証明書を渡す
JWTトークン
⬇ 「このトークンを持ってAPIを叩く」
🚪
3. トークンの検証 API Gateway が「偽物じゃないか」を確認
⬇
🧠
4. 本人データの取得 Lambda が「Aさんのデータだけ」をDynamoDBから取得

この構成の素晴らしいところは、「他人のデータは絶対に取れない」「アプリ内にAWSのアクセスキーを埋め込まなくて良い」という点です。

3. IAMとCognitoの違い（超重要）

「AWSにアクセスするならIAMユーザーを作ればいいのでは？」と思うかもしれませんが、それは危険です。
この2つは「誰向けか」が根本的に違います。

観点 AWS IAM Amazon Cognito
対象者 運営側 (管理者・開発者) 利用者側 (アプリの一般ユーザー)
利用人数 少数 (数十人〜数百人程度) 不特定多数 (数万人〜数千万人)
認証情報 アクセスキー (長期・漏れると危険) 一時トークン (数時間で失効)
用途 AWSリソースの作成や削除など 自社アプリへのログイン

アプリのユーザーが1万人いるからといって、IAMユーザーを1万人分作るのは設計として完全に間違っています（管理できませんし、権限が強すぎます）。一般ユーザーの管理はCognitoの仕事です。

4. なぜ分かりにくいのか？

Cognitoを勉強していると、STS（Security Token Service）や、JWT（JSON Web Token）、IAMロールといった別の技術要素がたくさん絡んでくるため、難しく感じます。
しかし、本質はシンプルです。

  • Cognitoは「パスワード管理を自分でやらないためのサービス」。
  • ログインに成功したら「一時的な通行証」をもらえる。
  • その通行証を使えば、AWSの他のサービス（S3やAPI Gateway）を安全に使える。

まとめ

Amazon Cognitoは、アプリ利用者向けの認証基盤であり、ログイン機能を提供しつつ、安全にAWSリソースへアクセスさせるためのサービスです。
試験で「モバイルアプリのユーザー認証」「何百万人ものユーザー」「ソーシャルログイン（Google連携など）」といったキーワードが出たら、Cognitoを一発で選びましょう！
