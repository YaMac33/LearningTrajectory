SQSによるバッチ処理の最適化

#SQS #EC2 #バッチ処理 #並列処理 #AWS #負荷分散

SQSを使ってEC2でのバッチ処理を並列化し、効率化する方法を解説します。

システム開発において、「重たいバッチ処理（画像変換や集計など）をどうやって効率よく終わらせるか」は永遠の課題です。
AWSでは、Amazon SQS（メッセージキュー）を使うことで、この処理を複数のEC2で手軽に「並列化・分散」させることができます。

1. アーキテクチャの比較（なぜSQSが必要？）

ユーザーが大量の画像をアップロードし、サーバーで「リサイズ・圧縮処理」を行うケースを考えてみましょう。

バッチ処理の設計比較
❌ 1台で頑張る設計
⬇

EC2

・SQSが「タスクの順番待ち箱」になる。
・空いているEC2からタスクを取って並列処理。
・EC2が落ちてもSQSにタスクが残る！

SQSのキューには、「画像Aをリサイズしてね」「画像Bをリサイズしてね」というメッセージ（タスクの指示書）が入ります。EC2たちはそれを1件ずつ取得し、処理が終わったらメッセージを削除します。

2. 具体的な活用事例

事例①：夜間バッチ処理（ECサイト）

  • 内容: 毎晩行う「売上集計」「在庫再計算」「レコメンド更新」。
  • 効果: 1台でやると2時間かかる1万件の注文データ処理も、SQSに入れてEC2を10台用意すれば、10倍速（わずか12分）で終わります。

事例②：動画エンコード

  • 内容: ユーザーがアップロードした動画を、1080p, 720p, 480p の各画質に変換する処理。
  • 効果: 「動画A」「動画B」という重いタスクをSQSでキュー化し、複数のEC2で同時並行でエンコードできます。

3. Auto Scalingと組み合わせると「最強」

さらに、SQSとAuto Scalingを連携させると、コストと速度の究極の最適化が可能になります。

Auto Scalingとの最強コンボ
📦 SQS メッセージが増える
➡
📈 Auto Scaling 「忙しい！」と検知
➡
💻 EC2追加 ワーカー台数を増やす
➡
✨ 処理速度UP キューが減ればEC2も減る

仕事が溜まっている時だけ一時的にEC2をドカッと増やし、終わったら減らす（スケールインする）ことで、無駄なサーバー代を払わずに済みます。

4. SQSがないとどうなる？（なぜSQSが必要か）

「EC2同士で直接通信して並列処理すればいいのでは？」と思うかもしれませんが、それは地獄の始まりです。

  • 競合の問題: 複数台のEC2が「同じ画像」を同時に処理してしまうのを防ぐ制御が極めて複雑になります。（SQSなら、誰かが処理中のメッセージは一時的に他から見えなくなる機能が備わっています）。
  • 障害時のロスト: 処理中にEC2がクラッシュすると、そのタスクは消えてしまいます。（SQSなら、一定時間完了報告がなければ、タスクをキューに戻して別の人にやらせます）。

5. 試験対策と実務のまとめ

出題キーワード 正解となるサービス / 構成
非同期処理、コンポーネントの分離 Amazon SQS (デカップリング)
並列バッチ処理、負荷分散 SQS + EC2 (ワーカー)
キューの長さに応じた自動スケール SQS + Auto Scaling

【実務での一歩先】
現代のAWSでは、ワーカーとしてEC2の代わりに AWS Lambda（サーバーレス） や ECS/Fargate（コンテナ） を使う構成が主流になっています。さらに大規模な分析処理なら AWS Batch という専用サービスもあります。

結論： SQSは「タスクの順番待ち箱」。EC2を増やせば、その箱から同時に取り出して並列処理できる！
