AWS OpsWorks の基本

#AWS #OpsWorks #構成管理 #Chef #Puppet

AWS OpsWorks は、サーバーの設定やアプリケーションのデプロイを自動化するための「構成管理サービス」です。

一言で言うと、「Chef（シェフ）や Puppet（パペット）という有名ツールを使って、サーバーの中身（設定）を自動で料理してくれるサービス」です。

試験対策としては、「Chef」「Puppet」という単語が出たら、反射的に「OpsWorks！」と答えられるようにするのが第一歩です。

1. なぜ OpsWorks が必要なのか？

CloudFormationは「EC2を作る（インフラ構築）」のは得意ですが、「EC2の中にApacheを入れて、設定ファイルを書き換えて、ユーザーを追加して…」というOS内部の細かい設定までは少し苦手です（User Dataでも書けますが、複雑になると管理が大変です）。

そこで、「構成管理ツール（Chef / Puppet）」の出番です。これらは「設定の手順書（レシピ）」を書くと、その通りにサーバーの中身を整えてくれます。OpsWorksは、このChefやPuppetをAWS上で使いやすくしたマネージドサービスです。

2. OpsWorks のイメージ： 「自動調理レストラン」

OpsWorksの仕組みは、レストラン運営に例えると非常にわかりやすいです。

OpsWorks 特有の「3階層構造」

  • スタック (Stack): 一番外側の枠組み。「本番環境」「開発環境」といったシステムの全体集合です。
  • レイヤー (Layer): ★ここが一番重要。
    サーバーの「役割」ごとのグループです。「Webレイヤーにはこのレシピ（手順）を適用する」「DBレイヤーにはあのレシピを適用する」と定義します。
  • インスタンス (Instance): 実際に動くEC2サーバーです。必ずどこかのレイヤーに所属します。

3. OpsWorks のすごい機能：「ライフサイクルイベント」

OpsWorksは、サーバーの状態変化に合わせて、勝手にレシピを実行してくれます。これをライフサイクルイベントと呼びます。

ライフサイクルイベントの流れ
Setup
起動時
→
Configure
構成変更時
→
Deploy
アプリ配置時
  • Setup（セットアップ）: サーバーが起動した瞬間に走ります。「Apacheのインストール」などはここでやります。
  • Configure（構成変更）: ★これがOpsWorksの真骨頂。
    「隣のサーバーが増減した時」に、生きている全サーバーで実行されます。
    例: Webサーバーが1台増えたら、ロードバランサーのレイヤーに対して「設定ファイルを書き換えて、新しいWebサーバーを振り分け先に追加して！」と自動で指令が飛びます。
  • Deploy（デプロイ）: アプリケーションのコード（PHPやRubyなど）を配置する時に走ります。
  • Shutdown（シャットダウン）: サーバーが停止する直前に走ります。「ログの退避」や「ロードバランサーからの切り離し」を行います。

4. CloudFormation との違い（試験対策）

どちらも「コードで自動化」ですが、守備範囲が違います。

特徴 CloudFormation OpsWorks
主な役割 インフラの構築
(AWSリソース作成)		 OS内部の設定
(ソフトのインストール)
使う言語 JSON / YAML
(AWS独自の形式)		 Chef / Puppet
(一般的な構成管理ツール)
得意分野 「VPCを作って、EC2を立てる」まで 「EC2の中にミドルウェアを入れて設定する」
キーワード テンプレート、スタック、IaC レシピ、クックブック、レイヤー

まとめ

  • 役割: Chef や Puppet を使って、サーバー内部の設定（構成管理）を自動化するサービス。
  • 構造: スタック（全体） > レイヤー（役割） > インスタンス（サーバー）。
  • 機能: ライフサイクルイベント（Setup, Configure等）を使って、サーバーの増減に合わせて設定を自動更新できる。

【補足：現代のAWS事情】
実は現在、OpsWorksは少し古いサービスになりつつあります。今は、AWS Systems Manager (SSM) というサービスでOS内部の設定管理を行うのが主流です。しかし、試験にはまだ「Chefを使いたい場合のサービスは？」としてOpsWorksが出題されますので、知識としては必須です。

これで、自動化ツールの「構成管理」担当であるOpsWorksもクリアです。
