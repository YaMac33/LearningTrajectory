AWS AppConfigの概要と利点

#AWS #AppConfig #設定管理 #機能フラグ #デプロイ戦略

AWS AppConfigはアプリの設定を安全に変更できるサービス。再デプロイなしで機能を制御！

AWS AppConfig（アップコンフィグ）は、AWS Systems Manager（SSM）の一機能で、「アプリケーションの『設定（コンフィグ）』だけを、コードを書き換えずに、安全かつ徐々に変更するためのサービス」です。
一言で言うと、「アプリの再起動やデプロイ（更新）をせずに、機能をオン・オフしたり、挙動を変えたりできる『リモコン』」です。

1. なぜこれが必要なの？（抱えている課題）

通常、アプリの挙動（例：割引率を10%から20%に変える、新機能を公開するなど）を変えるには、プログラムコードを修正して、ビルドして、サーバーにデプロイ（再配置）し直す必要があります。

  • リスク: デプロイ作業中にミスして、アプリ全体が止まるかもしれません。
  • 手間: たかが数字一つ変えるのに、何十分もかかります。

2. AppConfig の解決策：「コードと設定の分離」

AppConfigを使うと、「プログラム本体（コード）」と「設定値（コンフィグ）」を切り離して管理できます。

3. AppConfig だけが持つ「3つの安全装置」

ただ設定を変えるだけなら「Parameter Store」や「環境変数」でもできます。AppConfigが凄いのは、「設定ミスでアプリが壊れるのを防ぐ機能」がついている点です。

事故を防ぐ3つの仕組み
🛡️ ① バリデーター 事前チェック
「文法は正しいか？」「数字は0〜100の間か？」を自動確認し、ミスがあればブロック。
🐌 ② デプロイ戦略 徐々に公開
「最初は10%のユーザーだけ」→「問題なければ50%」と、時間をかけてじわじわ適用。
↩️ ③ 自動ロールバック 緊急停止
変更後にエラー率（CloudWatch）が急増したら、瞬時に前の設定に戻す。

4. 機能フラグ（Feature Flags）

AppConfigの最も有名な使い方がこれです。コードの中に「新機能A」を埋め込んでおき、AppConfigで「新機能A = OFF」にしてリリースします。

  • リリース日は、コードをいじらず、AppConfigのスイッチを「ON」にするだけで新機能が有効になります。
  • もしバグが見つかったら、すぐにスイッチを「OFF」にすれば、再デプロイなしで元通りです。

5. Parameter Store との違い（試験のポイント）

どちらも「設定値を保存する」サービスですが、役割が違います。

まとめ

  • AWS AppConfig: アプリの再デプロイなしで、設定や機能を動的に変更するツール。
  • メリット: 「バリデーション（チェック）」と「段階的デプロイ」で、設定変更による事故を防ぐ。

試験で「アプリケーションのコードを変更・デプロイすることなく、本番環境で機能を有効化/無効化したい。かつ、設定ミスによるダウンタイムを最小限にしたい」と聞かれたら、AppConfig が正解です！
