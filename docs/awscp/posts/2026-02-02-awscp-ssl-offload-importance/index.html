SSLオフロードの重要性

  • SSL Offload
  • AWS
  • ACM
  • ALB
  • インフラ
  • 負荷軽減

AWS認定講師として、インフラ構成を理解する上で非常に重要な概念である「SSLオフロード」について解説します。ACM（証明書管理）とALB（ロードバランサー）をセットで使う際、この「SSLオフロード」という仕組みが裏側で動いています。

1. SSLオフロードとは？

一言でいうと、「本来サーバーが行うべき『暗号化・復号』の重い作業を、ロードバランサーなどの別の機械に肩代わりさせること」です。

  • オフロード (Offload)： 荷物（負荷）を降ろす、という意味です。

例え話：ホテルの「コンシェルジュ」

オフロードなし：
客（通信）が部屋（サーバー）の前に来るたびに、サーバーが自分で鍵を開け、本人確認をし、荷物を検査します。これではサーバーが忙しくて本来の仕事（料理を出すなど）に集中できません。

オフロードあり（ALB＋ACM）：
ホテルの入り口に「コンシェルジュ（ALB）」を置きます。ここで鍵を開け、本人確認を済ませてしまいます。そこから部屋（サーバー）までは、安全な内線通路を通るだけなので、サーバーは鍵開けの手間がなくなり、本来の仕事に専念できます。

2. SSLオフロードの仕組み

図で見ると、データの「状態」が変化しているのがわかります。ALBが「暗号化の終点（SSL Termination）」となります。

ALBで暗号を解き、後ろのサーバーへは軽いHTTPで渡す。

3. SSLオフロードの「3つのメリット」

なぜわざわざこんなことをするのか、試験でも問われるメリットです。

📉 サーバー負荷軽減

暗号化・復号には計算パワーを使います。これをALBに任せることで、EC2サーバーのCPUを節約できます。

🔧 管理の一元化

100台のサーバーがあっても、証明書をインストールするのはALBの1箇所だけで済みます。ACMでの更新も1回で終わります。

👮 WAFとの相性

WAFは通信の中身を見て悪いやつをブロックします。ALBで一度「暗号を解く」からこそ、WAFが中身をチェックできるようになります。

4. 試験で問われやすいポイント（CLF-C02）

  • 設置場所： SSL/TLS証明書（ACM）をどこに配置するか？ → ALB（Application Load Balancer）やCloudFrontに配置して、そこで終端（オフロード）させます。
  • セキュリティと効率の両立： 「インターネット上は暗号化し、VPC内部（ALBから背後）は負荷を下げて効率化する」という考え方が正解の選択肢になりやすいです。
  • なぜ平文でいいの？： ALBの後ろ側は、AWSの専用ネットワーク（VPC）内です。ここは部外者が立ち入れない安全な場所なので、あえて暗号化を解いたまま（HTTP）でサーバーに渡しても安全とされています。

5. 実務イメージ：大規模なキャンペーン

1分間に数万件のアクセスが来るキャンペーンサイト。すべてのアクセスをサーバー側で暗号解除していると、CPUが100%になってサイトがダウンしてしまいます。

そこでACMをセットしたALBでSSLオフロードを行うことで、サーバーは「純粋なリクエスト処理」だけに集中でき、少ない台数のサーバーで大量のアクセスをさばけるようになります。

まとめ

「暗号解除という面倒な作業を入り口（ALB）で済ませて、中（サーバー）を楽にさせてあげること」がSSLオフロードです。「SSLオフロード」という言葉を聞いたら、ALBでカギを開けて、中身を裸にしてからサーバーに渡す図を思い出してください。
