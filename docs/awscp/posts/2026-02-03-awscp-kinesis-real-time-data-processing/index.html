Kinesisによるリアルタイムデータ処理

#Kinesis #データ処理 #ストリーミング #AWS #リアルタイム

Amazon Kinesis（キネシス）は、「大量のデータが、絶え間なく流れ込んでくる状況（ストリーミングデータ）」をリアルタイムで処理するためのサービス群です。

一言で言うと、「情報の激流を受け止める『土管』や『ダム』のようなインフラ」です。IoTセンサー、Webサイトのクリックログ、SNSの投稿など、「止まることなく流れ続けるデータ」を扱うのに特化しています。

1. なぜ Kinesis が必要なのか？（バッチ vs ストリーム）

これまでのデータ処理は「バケツがいっぱいになったら運ぶ（バッチ処理）」のが普通でした。しかし、「クレジットカードの不正利用」や「株価の急変動」など、**「1秒でも早く気づきたい」**というシーンでは、バケツがいっぱいになるのを待っていられません。

Kinesisは、データが発生した瞬間に**「流れながら処理する（ストリーム処理）」**ことを可能にします。

データ処理方式の違い
🪣
バッチ処理
バケツに溜めてから運ぶ。
効率は良いが、
結果が出るまで遅い。
🚿
ストリーム処理 (Kinesis)
ホースで流し続ける。
発生した瞬間に処理でき、
即座に反応できる。

2. Kinesis ファミリーの「4兄弟」

Kinesisは単一のサービスではなく、役割の違う4つのサービスの総称です。試験では特に**上から2つ（StreamsとFirehose）**の違いが問われます。

Streams と Firehose の違い
🛣️ Data Streams 手動管理
  • 役割：情報の「土管」
  • シャード（車線）の数を自分で管理
  • データは一時保存され、後からリプレイ可能
  • 用途：複雑なリアルタイム分析、カスタムアプリ
🚒 Data Firehose 全自動
  • 役割：S3への「配送業者」
  • 管理不要（フルマネージド）
  • データを変換してS3 / Redshiftへ保存
  • 用途：とにかくログを保存したい時

③ Kinesis Data Analytics (KDA)

  • 役割： 流れているデータに対して、「SQL」を使ってリアルタイムに集計・フィルタリングができる機能です。
  • 用途： 「エラーログが1分間に10件流れてきたらアラートを出す」といった即時対応に使います。

④ Kinesis Video Streams (KVS)

  • 役割： 監視カメラやスマホからの「映像・音声」データをリアルタイムでAWSに送るサービスです。

3. SQS と Kinesis の違い（試験の超・頻出ポイント）

「データを一時的に溜めて処理する」という点で Amazon SQS と似ていますが、使い道が全く違います。

特徴 Amazon SQS (キュー) Amazon Kinesis (ストリーム)
たとえ 「郵便ポスト」 「ベルトコンベア / 防犯カメラ録画」
データの単位 個別の「メッセージ」 (手紙) 連続した「レコード」 (映像/ログ)
処理後 消える (取り出したら削除) 消えない (保存期間内なら何度でも読める)
読み手 基本的に1つのシステム 複数のシステムが同時に読める
用途 タスクの分散、システム連携 ビッグデータ収集、ログ解析

試験の合言葉:
・タスクを1つずつ確実に処理したい → SQS
・ログをS3に保存したい、リアルタイム分析したい → Kinesis

4. 試験における「ストリーミングデータ」の正体

過去問などで「ストリーミングデータ」という言葉が出た時、Video Streams (KVS) と混同しないように注意が必要です。

  • ✕ 動画のことだけではない: NetflixやYouTubeのような映像データだけを指すわけではありません。
  • ◎ 連続データ全般を指す: Webサイトのクリックログ、株価、IoTセンサーの温度データなど、「絶え間なく発生する文字データ」もストリーミングデータです。

もし試験で「ストリーミングデータを収集・分析する」とあったら、映像という言葉がない限り、基本的には Kinesis Data Streams のことを指しています。

まとめ

  • Kinesis Data Streams: 「リアルタイム」重視。シャード管理が必要。データはリプレイ可能。
  • Kinesis Data Firehose: 「S3への保存」重視。管理不要で楽ちん。S3/Redshiftへのロードといえばこれ。
  • 見分け方: 「ログをS3に保存したい」ならFirehose。「リアルタイムにカスタム処理したい」ならStreams。
