AWSのバッチ処理とETLツール

#AWS #バッチ処理 #ETL #Glue #EMR

データのバッチ処理を行うAWS GlueとAmazon EMRの違いを解説します。

「データのバッチ処理」とは、Kinesisのような「リアルタイム（その場ですぐ）」な処理とは対極にある、「ある程度データを溜め込んでから、あとでまとめて一気に処理する方法」のことです。
例えば、「1ヶ月分の売上データを集計してレポートを作る」とか、「夜中の間に昨日の全データを分析する」といった作業がこれに当たります。

AWSでこの「まとめてドカンと処理（バッチ処理）」をする際の2大巨塔が、AWS Glue と Amazon EMR です。それぞれの役割と違いを、わかりやすい例えで解説します。

バッチ処理の2大巨塔：役割の違い
🧹🤖
AWS Glue
全自動お掃除ロボット
  • 目的: データの整理整頓 (ETL)
  • 特徴: サーバーレス (管理不要)
  • 得意技: データカタログ作成
👨‍🔬💻
Amazon EMR
計算専門の科学者チーム
  • 目的: 高度な計算・分析
  • 特徴: クラスター (Hadoop/Spark)
  • 得意技: ビッグデータ解析

1. AWS Glue（グルー）

キャッチコピー： 「全自動のデータお掃除ロボット（サーバーレス ETL）」

データ分析の準備（ETL）に特化しています。
イメージは、散らかった部屋（S3にある生データ）から、ゴミを捨て、整理整頓して、タンス（データベース）にきれいにしまう「全自動お掃除ロボット」です。

  • サーバーレス: ロボット（サーバー）の管理は不要です。「毎晩3時に掃除して」と命令するだけで勝手に動いてくれます。
  • データカタログ: 「どこにどんなデータがあるか」という「在庫リスト（カタログ）」を自動で作ってくれる機能が超優秀です。
  • ETLが得意: 「Extract（抽出）」「Transform（変換）」「Load（書き出し）」の略で、データの整形・加工に向いています。

向いているバッチ処理

  • S3にあるCSVファイルを読み込んで、不要な列を削除し、Redshift（分析用DB）に入れる。
  • 毎日決まった時間に、データの形式を変換する（JSON → Parquetなど）。

2. Amazon EMR（イーエムアール）

キャッチコピー： 「超高性能な計算専門の科学者チーム（Hadoop / Spark）」

大規模なデータの高度な計算・分析に特化しています。
イメージは、難しい数式や大量の統計データを処理するために、何台ものパソコンを繋いで作った「スーパーコンピューター（クラスター）」です。

  • オープンソース: Hadoop（ハドゥープ）やSpark（スパーク）といった、世界中で使われているビッグデータ処理ツールをAWS上で簡単に動かせます。
  • カスタマイズ: どんなスペックのパソコン（EC2）を何台使うか、自分で細かく設計できます。
  • パワー: Glueよりもさらに複雑で重たい計算（機械学習や科学技術計算）ができます。

向いているバッチ処理

  • 数億件の顧客データを使って、機械学習のモデルを作る。
  • 過去10年分のログデータを解析して、複雑なトレンド分析をする。

3. Glue と EMR の決定的な違い（試験・実務用）

どちらも「バッチ処理」に使えますが、使い分けのポイントはここです。

特徴 AWS Glue Amazon EMR
主な目的 ETL (データの移動・整理) ビッグデータ解析 (計算)
管理の手間 サーバーレス (管理不要で楽ちん) クラスター (EC2の管理が必要)
キーワード 「データカタログ」「ETL」「サーバーレス」 「Hadoop」「Spark」「ビッグデータ」
コスト 動いた時間だけ課金 (準備不要) 常に起動しておくか、起動・停止の管理が必要
たとえ 「お掃除ロボット」 「スーパーコンピューター」

まとめ

  • データのバッチ処理 = 溜まったデータを後でまとめて処理すること。
  • AWS Glue = データをきれいに整えて、分析しやすい場所に運ぶための「便利な自動ツール」。
  • Amazon EMR = 膨大なデータをバリバリ計算するための「強力な処理エンジン」。

試験で「ETL処理をサーバーレスで行いたい」と出たら Glue。「HadoopやSparkを使ってビッグデータを処理したい」と出たら EMR を選んでください。
