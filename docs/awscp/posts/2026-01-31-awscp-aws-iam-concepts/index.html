AWS IAMの基本概念

  • AWS
  • IAM
  • セキュリティ
  • 管理
  • クラウド

AWSを「城」に例えるなら、IAM（アイアム：Identity and Access Management）は、城門に立つ「門番」であり、城内の「通行手形発行所」です。ここを適当に済ませると、せっかく作った最強のインフラも、ハッカーに乗っ取られて終わりです。試験でも実務でも最重要項目です。

1. IAMの役割：認証と認可

IAMは大きく分けて2つの仕事をします。

  • 認証 (Authentication): 「あなたは誰？」を確認する（ログイン）。
  • 認可 (Authorization): 「あなたは何をしていい人？」を確認する（権限）。

2. IAMの「登場人物」4選

IAMの概念は、一般的な「会社組織」に当てはめると一発で理解できます。

IAMの用語 会社の用語 説明
IAM ユーザー 社員 個人に発行されるID。「Aさん」「Bさん」。
IAM グループ 部署 「営業部」「開発部」。ユーザーをまとめて管理する箱。
IAM ポリシー 辞令・就業規則 「S3を見てよし」「EC2を消してよし」という許可証（ルールブック）。
IAM ロール 帽子・腕章 「工事作業員」のような役割（帽子）。これを被っている間だけ、その権限を使える。人だけでなく、EC2などの機械も被れる。

3. 図解：権限を与える仕組み

AWSでは、「ユーザー」に直接「ポリシー（許可証）」を渡すこともできますが、人数が増えると管理が大変です。基本は「グループにポリシーを貼り付け、そこにユーザーを入れる」という運用をします。

IAM ポリシー (S3フルアクセス許可)
↓ 貼り付け (アタッチ)
📁 IAM グループ (開発チーム)
👤 Aさん
👤 Bさん

Result: AさんもBさんも、S3を自由に触れるようになります。

4. 試験の最難関「IAM ロール」とは？

「ユーザー」と「ロール」の違いが試験でよく問われます。

  • IAM ユーザー： 「Aさん」という特定の人のための永続的なID。パスワードを持つ。
  • IAM ロール： 特定の人に紐付かない「一時的な権限の貸し出し」。パスワードはない。

いつ使うの？（試験に出る2大パターン）

① EC2インスタンスに権限を与えたい時

EC2上のプログラムがS3のデータを読みたい時、EC2の中にIDとパスワード（アクセスキー）を保存するのはセキュリティ的に最悪です（流出リスクがあるため）。代わりに、EC2に「S3閲覧ロール（帽子）」を被せます。

🤠
EC2
(サーバー)
➡ アクセスOK!
S3

EC2は「帽子」をかぶっている間だけ、S3への通行手形を持てます。

② 別のアカウントの人が一時的に操作したい時

「関連会社のCさんに、今日だけ作業を手伝ってほしい」という時、CさんにIDを発行するのではなく、「作業用ロール」を作って、それを被ってもらいます（スイッチロールと言います）。

5. 試験に出る「3つの鉄の掟」

AWSのセキュリティ問題は、以下の思想に基づいて正解を選べば9割当たります。

  • ① ルートユーザーは封印せよ
    ルートユーザーはAWSアカウントを作った時の「神」アカウントです。何でもできるし、消せます。最初に管理者用の「IAMユーザー」を作ったら、ルートユーザーは金庫にしまって二度と使わないのが正解です。
  • ② MFA（多要素認証）を有効化せよ
    パスワードだけでは漏洩します。「スマホアプリのワンタイムパスワード」などを組み合わせるMFAを、特にルートユーザーには絶対に設定します。
  • ③ 最小権限の原則 (Least Privilege)
    「その作業に必要な最低限の権限だけを与えること」です。S3を見るだけのバイト君に、EC2を削除する権限を与えてはいけません。試験で「全権限（AdministratorAccess）」を与える選択肢は、基本的に不正解です。

6. まとめ

  • IAMユーザー = 社員ID
  • IAMグループ = 部署
  • IAMポリシー = 許可証（JSON形式）
  • IAMロール = EC2などの機械や、一時的な利用者に貸す「帽子」

これで、AWSという城の中に入るためのルール（IAM）が分かりました。
