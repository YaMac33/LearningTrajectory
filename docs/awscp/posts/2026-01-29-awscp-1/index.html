IAM（アイデンティティ管理）

  • IAM
  • AWS
  • アクセス管理
  • セキュリティ
  • ポリシー

IAM（Identity and Access Management）は、AWSを利用する上で最も基本的かつ重要なサービスです。一言で言えば、「誰が・何を・どこまでAWSでできるか」を厳密に制御する仕組みです。

1. IAMとは何か

  • 認証・認可の管理： Amazon Web Services（AWS）上での「誰が（認証）」、「何をしてもよいか（認可）」を管理します。
  • アクセス制御： AWSリソースへのアクセスを、ユーザー、役割、ポリシーを用いてきめ細かく制御します。
  • グローバルサービス： IAMはリージョン（東京、バージニアなど）に依存せず、AWSアカウント全体で共通の設定として機能します。

2. IAMで管理する3つの主役（重要）

IAMを理解するためには、以下の3つの要素（プリンシパル）を区別することが重要です。

① ユーザー（User）

  • 人やシステムを表すアカウント： 特定の人物やアプリケーションに紐づくIDです。
  • 操作の主体： マネジメントコンソールへのログインや、API操作を行う主体となります。
  • 運用のポイント： 本番環境の運用では、個人ごとのIAMユーザーを直接使わず、後述する「ロール」を使用することも一般的です。

② グループ（Group）

  • ユーザーの集合： 複数のIAMユーザーをまとめる箱です。
  • 権限の一括管理： 「Adminグループ」「ReadOnlyグループ」のように役割ごとにグループを作り、そこに権限（ポリシー）を付与することで、管理の手間を減らします。

③ ロール（Role）【試験頻出】

  • 一時的な権限の帽子： ユーザーやAWSサービスが「一時的に」かぶることができる役割です。
  • 資格情報を持たない： 固定のパスワードやアクセスキーを持ちません。
  • 主な用途：
    • EC2 → S3： EC2インスタンスがS3バケットのファイルを読み書きする場合。
    • Lambda → DynamoDB： Lambda関数がデータベースを操作する場合。
    • クロスアカウントアクセス： 他のAWSアカウントから自社環境へアクセスさせる場合。

3. ポリシー（Policy）とは

ポリシーはIAMの心臓部であり、「何を許可し、何を拒否するか」を定義したルールセットです。通常はJSON形式で記述されます。

ポリシーの基本構造

  • Effect： Allow（許可） または Deny（拒否）
  • Action： 具体的に何をするか（例：s3:GetObject ならファイルの取得）
  • Resource： どのリソースに対して行うか（例：特定のS3バケット）

重要ルール： AWSの権限評価では、「明示的なDeny（拒否）」が最優先されます。AllowがあってもDenyがあれば拒否されます。

5. ユーザー vs ロール（試験対策）

AWS認定試験で頻出の「ユーザー」と「ロール」の違いを比較表にまとめました。AWSのベストプラクティスは「可能な限りロール（一時的な認証情報）を使うこと」です。

観点 ユーザー (User) ロール (Role)
主体 人・固定のアカウント 一時的な役割・権限セット
資格情報 あり（パスワード / アクセスキー） なし（一時的なトークンが発行される）
主な用途 管理者、開発者のログイン EC2、Lambda、外部連携
セキュリティ キー漏洩のリスクが常にある 時間が来ると無効になるため安全性が高い

6. ルートユーザー（Root User）【要注意】

AWSアカウントを作成した最初に存在するユーザーを「ルートユーザー」と呼びます。これは全能の神のような権限を持っています。

  • IAMでは制限不可： IAMポリシーでルートユーザーの権限を制限することはできません。
  • 原則封印： 普段の操作には絶対に使用せず、緊急時や請求関連の操作のみに使用します。

必須のセキュリティ対策

  1. MFA（多要素認証）の有効化： スマホアプリやハードウェアトークンを設定する。
  2. アクセスキーを作らない： プログラムからの操作に使わない。

7. AWS CP試験でのひっかけポイント

Cloud Practitioner試験などでよく問われる間違いやすいポイントです。

  • ❌ IAMはリージョンごとのサービスである → ⭕ グローバルサービスです。設定は全リージョンに反映されます。
  • ❌ EC2インスタンスの中にIAMユーザーのアクセスキーを保存してアクセスする → ⭕ IAMロールをEC2に割り当ててアクセスします（キーを置くのはNG）。
  • ❌ 権限がないのは「Allow」の設定がないからだけである → ⭕ 「Allowがない（暗黙の拒否）」場合もありますが、「明示的なDeny」がある場合は、どんなに強い権限があっても拒否されます。

8. まとめ

IAMを一言で覚えるなら、AWSの「入館証（認証）＋職務権限表（認可）」です。セキュリティの要となるサービスですので、まずは「ユーザー・グループ・ロール・ポリシー」の4つの関係性をしっかり押さえておきましょう。
