EMRとデータレイクの正しい理解

#データレイク #EMR #Redshift #AWS #データ処理

EMRはデータレイクではなく加工エンジン。データの貯蔵と処理を正しく分けよう。

Q. EMRは「データレイク」、Redshiftは「DWH」という認識で合ってる？
A. その認識は「半分正解で、半分は少しだけ修正が必要」です！

非常に惜しいところですので、試験で迷わないように「データレイク」という言葉の定義と、各サービスの役割分担を整理しましょう。

1. 正しい位置づけ：3つの役割

実は、AWSにおいて「データレイク（貯める場所）」という役割を担うのは、EMRではなく Amazon S3 なのです。

データ分析基盤の3大要素
🌊 Amazon S3 データレイク
何でも放り込める
「巨大な湖」。
(貯蔵)
⚙️ Amazon EMR 加工エンジン
水を汲み上げて
「ろ過・加工する工場」。
(計算)
🏙️ Amazon Redshift DWH
整理された水が入った
「巨大な貯水タンク」。
(分析)

実務でのデータフロー

これらは以下のように連携して動きます。

データの流れ
S3
(Raw)
読み込み ➡
EMR
(加工)
書き戻し ➡

(Clean)
ロード ➡

(分析)
  1. S3 (データレイク): ログ、画像、CSVなど、あらゆる生データをそのまま貯めます。
  2. EMR (加工): S3にある生データを取り出し、不要なデータを削除したり形を整えたりして、またS3に戻します。
  3. Redshift (DWH): 加工済みのきれいなデータを溜め込み、ビジネス分析（BIツールなど）に利用します。

2. なぜ「EMR ＝ データレイク」と混同しやすいのか？

それは、EMRが「データレイク（S3）にあるデータを直接処理するのが得意だから」です。
昔のビッグデータ技術（オンプレミスのHadoopなど）では、データを溜める場所（HDFS）と処理する場所が一体化していました。
しかしAWSでは、「貯める場所（S3）」と「計算する場所（EMR）」を切り離して考えるのが鉄則です。これを「ストレージとコンピューティングの分離」と呼び、コスト削減の大きな鍵となります。

3. Redshift は間違いなく「DWH」です

こちらはあなたの認識通り、100%正解です！
Redshiftは、数ペタバイトという途方もない量の「整理されたデータ（構造化データ）」を、SQLを使って高速に検索するための専用サービスです。

4. 試験で問われやすいポイント

試験問題では、以下のようなキーワードでサービスを出し分けてきます。

要件・キーワード 正解のサービス
「あらゆる形式のデータを安価に、無制限に貯めたい」
「データレイクを構築したい」		 Amazon S3
「HadoopやSparkなどのフレームワークを使いたい」
「大量のデータを加工・処理したい」		 Amazon EMR
「標準的なSQLを使って、大規模データの集計・分析を行いたい」
「DWH (データウェアハウス) が必要」		 Amazon Redshift
💡 講師のワンポイントアドバイス

最近の試験では Amazon Athena というサービスもよく出ます。
これは「データレイク（S3）にあるデータに対して、EMRやRedshiftを使わずに、直接SQLを投げて分析できる」サービスです。
「サーバーレスで手軽に分析」という言葉が出たら Athena を疑ってください。
