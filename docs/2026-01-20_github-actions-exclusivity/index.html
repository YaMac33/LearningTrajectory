GitHub Actionsの排他制御

GitHub Actionsは競合を検知し同時処理を停止するが、厳密には「排他制御」ではない。

1行要約：原則としてそれは「排他制御そのもの」ではなく、GitHub側が行っている競合回避（同時実行制限・競合検出）の結果である。

# 解説：トランザクションに関する排他制御とは

  • 定義 複数の処理（トランザクション）が同じデータを同時に更新しないよう制御する仕組み。
  • 目的 データ不整合の防止（同時更新・上書き・中途半端な状態を防ぐ）。
  • 典型例 データベースのロック（排他ロック）。
    BEGIN → 処理 → COMMIT / ROLLBACK

# GitHub Actionが1つ停止するケースの正体

起きていること

同じリポジトリ・同じブランチ・同じリソース（例：同一ファイル）に同時に書き込み処理が走った場合。

GitHub側の対応

  • 競合を検知して、後続のActionをキャンセル
  • 失敗扱い、もしくは待機させる
  • ＝ 安全側に倒して止めている
Process Flow 
処理A ──┐
        ├→ 同じGitHubリソースに書き込み
処理B ──┘
   ↓
GitHubが判断

処理A：継続
処理B：停止（競合回避）

# A vs B（ここが超重要）

では、これは排他制御に「該当するか」？試験的な正解としては、「トランザクションの排他制御とは言わない」が正解です。
以下にその違いを整理します。

観点 トランザクション排他制御 GitHub Actionの停止
主体 アプリ / DB GitHub（基盤側）
制御方法 ロックして待たせる 競合した処理を止める
データ単位 レコード・テーブル リポジトリ・ブランチ・ファイル
試験用語 排他制御 同時実行制限 / 競合回避

# ITパスポート・試験的な言い換え

排他制御

「同時更新させないよう順番待ちさせる」

GitHub Action停止

「同時実行を許可しない設計」

⚠️ もし選択肢に出たら

「GitHubに対して同時に処理を行った際、一方の処理が停止した。これは排他制御である。」

👉 ×（誤り）

排他制御は「アプリケーションやDBが明示的に行う制御」を指します。

補足（実務視点）

GitHub Actionsでも、concurrency、ロックファイル、キューイングなどの仕組みを使えば、排他制御“的”な設計は可能です。
（ただし試験ではそこまで問われません）
