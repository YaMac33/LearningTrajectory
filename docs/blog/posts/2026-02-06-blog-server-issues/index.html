サーバーが落ちるとは？

#サーバー #GitHub #障害 #アクセス集中 #技術

「サーバーが落ちる」という言葉をよく耳にしますが、具体的に内部で何が起きているのでしょうか。その原因と、GitHub Pagesのような静的ホスティングサービスの実情について解説します。

「サーバーが落ちる」状態と原因

一言でいうと、サーバーが処理能力の限界を超えてしまい、リクエストに対して正常に応答できなくなった状態を指します。利用者から見ると「ページが開かない」「503エラーが出る」といった現象になります。

主な原因は以下の4つに分類されます。

  • アクセス集中（過負荷）：チケット発売時などに同時アクセスが殺到し、CPUやメモリを使い切る。
  • プロセス停止：プログラムのバグやメモリリークで、Webサーバーソフト自体がクラッシュする。
  • インフラ障害：データセンターの停電や、物理的なネットワーク切断。
  • 意図的攻撃：DDoS攻撃など、悪意を持って大量の通信を送りつける行為。

GitHub Pagesは「サーバー」を使っていない？

「GitHub Pagesはサーバーを立てずに公開できる」と言われますが、これは「サーバーが存在しない」わけではありません。「ユーザーが管理すべきサーバーがない」だけで、裏側ではGitHubの巨大なインフラが動いています。

GitHub Pagesが強い理由

GitHub Pagesの実体は、CDN（コンテンツ・デリバリー・ネットワーク）に近い仕組みです。世界中に配置されたサーバーがあなたのWebサイトのコピー（キャッシュ）を持っており、ユーザーに最も近い場所からファイルを渡します。

  • 計算しない：データベース接続や複雑な計算を行わず、ファイルを渡すだけです。
  • 分散される：アクセスが集中しても、世界中のサーバーが分担して処理します。

そのため、個人のブログやポートフォリオレベルのアクセス集中で「落ちる」ことは、理論上ほぼありません。

「落ちない」代わりの制約

GitHub Pagesは強力ですが、その仕組み上、できないこともあります。

  • 動的な処理ができない：お問い合わせフォームの送信処理、ログイン機能、データベースへの保存などは単体ではできません。
  • リアルタイム生成不可：アクセスごとに表示内容を変える（例：検索結果の動的生成）ことは苦手です。

まとめ

「サーバーが落ちる」とは、処理能力オーバーや障害のことです。GitHub Pagesは「落ちない魔法」ではなく、「負荷がかかりにくい配信構造（CDN）」を採用しているため、結果として非常に安定しています。

AWSのS3 + CloudFrontのような構成もこれと同じ考え方です。「落ちないサイト」を作りたい場合、サーバーのスペックを上げるよりも、こうした「静的配信」の仕組みを活用するのが現代的なアプローチと言えます。
